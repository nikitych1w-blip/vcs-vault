# Page Object Model в проекте

## Общее описание

В проекте используется паттерн Page Object Model (POM) для организации тестов и UI-компонентов. Этот паттерн позволяет разделить логику взаимодействия с интерфейсом от логики тестирования, обеспечивая более чистую и поддерживаемую архитектуру.

## Структура проекта

### Папка UI

```
src/ui/
├── components/          # Компоненты интерфейса
│   ├── base.component.ts  # Базовый компонент
│   └── ... другие компоненты
└── pages/               # Страницы приложения
    ├── base.page.ts       # Базовая страница
    └── repo/              # Папка для страниц репозиториев
        └── settings.page.ts # Страница настроек репозитория
```

## Page Object Model

### Базовая страница (Base Page)

Файл: `src/ui/pages/base.page.ts`

Базовая страница предоставляет общие методы и свойства для всех страниц проекта. Она служит основой для создания специфичных страниц.

### Конкретные страницы

Каждая страница приложения реализуется как отдельный класс, наследующийся от базовой страницы. Пример:

Файл: `src/ui/pages/repo/settings.page.ts`

```typescript
import { Page } from '@playwright/test';

import List from '@vcs-pw/ui/components/list.component';
import BasePage from '@vcs-pw/ui/pages/base.page';

export default class RepoSettingsPage extends BasePage {
  readonly menu: List;

  constructor(readonly page: Page) {
    super(page, 'Репозиторий. Настройки', page.locator('.page-content.repository.settings'));

    this.menu = new List<Element>({
      locator: this.content.locator('.menu'),
      itemLocatorOrFactory: '.item',
    });
  }
}
```

## Рекомендации по использованию

1. **Используйте public readonly поля**: Для уменьшения дублирования и обеспечения доступа к элементам и компонентам из тестов
2. **Предпочтение композиции перед наследованием**: Компоненты создаются как зависимости, а не через наследование
3. **Определение локаторов только в классах страниц/компонентов**: Все селекторы определяются внутри соответствующих классов
4. **Используйте pageRegistry для удобного доступа к страницам**: Для централизованного доступа к классам страниц
5. **Для перехода по ссылке передавайте значения pathVariable**: При использовании метода goToEndpoint передавайте параметры в виде объекта контекста
6. **Используйте селекторы с data-testid**: Все элементы должны иметь атрибуты data-testid для удобного поиска
7. **Разделяйте ответственность**: Каждая страница отвечает за свою область функциональности
8. **Создавайте компоненты для часто используемых элементов**: Например, навигационное меню, кнопки, формы
