# Инструкция по использованию промта для автоматизации тестирования Frontend (UI/FE)

## Обзор

Данный промт (`promt_v1.txt`) предназначен для автоматизации полного цикла ручного тестирования frontend-сценариев через систему управления тестированием (TMS) с использованием агента Cline. Промт реализует 5-этапный процесс с обязательной валидацией на каждом шаге.

## Структура проекта

```
frontend_agent_v1/
├── promt_v1.txt              # Основной промт с инструкциями (это файл)
├── shablon.md                # Шаблон для тест-кейсов
├── test_results_shablon.json # Шаблон отчёта о тестировании
├── config.md                 # Конфигурация (учётные данные)
├── generate_vcs_data.py      # Скрипт генерации тестовых данных
├── import_tests_to_tms_full.py # Скрипт импорта в TMS
├── parse_xlsx.py             # Скрипт парсинга Excel из TMS
├── generate_test_cases.py    # Генератор TC из spec.md
└── vcs-knowledge/            # Документация и тестовые модели
```

## Требования

- Python 3.x (для скриптов импорта и генерации данных)
- Доступ к TMS: `https://portal.works.prod.sbt`
- Доступ к VCS Frontend: `https://sc-vp.ift.pd04.pvw.sbt/web/v2/`
- Учётные данные в файле `config.md`

- **Операция CRUD** (1% TC): "Создать / Удалить [объект]"

### Популярные паттерны результатов:
- **Отображение** (50%): "=> Отображается [элемент]"

### Формат шагов:
```
**Шаг N:** [Глагол действия] [объект]
**Тестовые данные:** [Если требуется]

```

## Этапы выполнения

### Этап 1: Анализ UI-узлов и спецификации

**Входные данные:**
- Test-model: `vcs-knowledge/vault/test-model/1. UI/`
- SA-требования: `vcs-knowledge/vault/specs/`
- UI-узел для тестирования (например: `UI / Проект / Репозитории / Репозиторий / Настройки / Защита веток`)

**Анализ:**
1. Изучить структуру UI-узла в test-model
2. Найти соответствующие AC в spec.md
3. Сопоставить AC с узлом test-model
4. Определить набор требуемых тест-кейсов ( lv8 - атомарные сценарии)
5. Определить версию API: web1 или web2

**Техники тест-дизайна (обязательно к использованию):**
1. ✅ Эквивалентное разделение (Equivalence Partitioning)
2. ✅ Граничные значения (Boundary Value Analysis)
3. ✅ Попарное тестирование (Pairwise Testing)
4. ✅ Предугадывание ошибок (Error Guessing)
5. ✅ Тестирование на основе состояний (State-Based Testing)

**Валидация:** Агент запросит подтверждение после анализа.

---

### Этап 2: Генерация тест-кейсов для UI

**Входные данные:**
- SA-требования (AC из spec.md)
- Test-model (UI-узлы)
- Паттерны шагов (из patterns.md)
- Техники тест-дизайна

**Формат вывода:**
- Использовать строго шаблон из файла `shablon.md`
- Имя файла: `[UI_NODE]_[VERSION].md` (пример: `Защита_веток_web2.md`)
- Каждый TC содержит:
  - id: TC-XXX
  - name: Название
  - priority: critical/major/minor
  - ui_node: UI-путь в test-model
  - api_version: web1 или web2
  - steps: Список шагов по паттернам

**Обязательные элементы для каждого шага:**
1. Использовать паттерны из `patterns.md`:
   - Авторизация: "Авторизоваться под [ролью]"
   - Навигация: "Перейти на [страницу]"
   - Кнопка: "Нажать [на кнопку]"
   - Форма: "Заполнить [поле]"
2. Результаты по паттернам:
   - Отображение: "=> Отображается [элемент]"
   - Успех: "=> [Элемент] успешно [действие]"

**Требования к тест-кейсам:**
1. **Один сценарий на тест-кейс:**
   - Каждый TC проверяет ОДИН конкретный сценарий
   - Не смешивать happy path и edge cases в одном TC

2. **Проверка UI-элементов:**
   - Указывать конкретные элементы (не "кнопка", а "кнопка [Название] в правом верхнем углу")
   - Проверять visibility, clickable, текст

3. **Версия UI обязательна:**
   - Указывать web1 или web2
   - Для web2: проверять SPA-навигацию, URL обновление
   - Для web2: проверять GraphQL запросы в Network tab

4. **Альтернативные сценарии обязательны:**
   - Пустые состояния (заглушка)
   - Ошибки валидации (сообщения об ошибках)
   - Edge cases (пустые значения, граничные условия)
   - Проверка состояния загрузки (лоадер)

**Пример TC по паттернам:**
```markdown
## TC-001: Защита веток - Успешное создание правила

**Id:** TC-001
**Название:** Защита веток - Успешное создание правила
**Приоритет:** critical
**Уровень теста:** ui
**Теги:** ui, branch, protection, web2, success

### Шаги:

**Шаг 1:** Авторизоваться под **Владельцем проекта**
**Тестовые данные:** admin_user
**Ожидаемый результат:** Успешная авторизация на платформе

**Шаг 2:** Перейти на вкладку **Проект** / Выбрать проект / Перейти в **Настройки** / **Защита веток**
**Тестовые данные:** project=agentvp
**Ожидаемый результат:** Открывается страница защиты веток

**Шаг 3:** Нажать кнопку **Добавить правило**
**Тестовые данные:** -
**Ожидаемый результат:** Открывается форма создания правила

**Шаг 4:** Заполнить поле **Паттерн ветки** значением **main**
**Тестовые данные:** main
**Ожидаемый результат:** Поле заполнено, паттерн отображается

**Шаг 5:** Нажать кнопку **Сохранить**
**Тестовые данные:** -
**Ожидаемый результат:** Правило создано, отображается в списке

**Шаг 6:** Проверить, что правило отображается в списке
**Тестовые данные:** -
**Ожидаемый результат:** Отображается правило с паттерном "main"
```

**Валидация:** Агент запросит подтверждение после генерации файла.

### Этап 3: Анализ структуры папок в TMS

**Действия агента:**
1. Выполняет POST запрос к API TMS для получения структуры папок
2. Определяет путь для новой папки:
   - Путь: `SC v2 → 1. UI → [название экрана] / [версия API] → [новая папка]`
   - Пример: `SC v2 → 1. UI → Защита веток / web2 → [новая папка]`
3. Проверяет существование папки

**Валидация:** Агент покажет путь и ID папки для подтверждения.

---

### Этап 4: Импорт тест-кейсов в TMS

**Параметры скрипта `import_tests_to_tms_full.py`:**

| Аргумент | Описание |
|----------|----------|
| `--create-cycle` | Флаг создания цикла тестов |
| `--username` | Логин пользователя (из config.md) |
| `--password` | Пароль пользователя (из config.md) |
| `--folder` | UUID папки (если существует) |
| `--folder-name` | Название папки: "[UI_NODE] [версия API]" |
| `--parent-folder` | ID родительской папки (если папка не существует) |
| `--cycle-name` | Название цикла тестов |
| `[путь к файлу]` | Путь к файлу с тест-кейсами |

**Предварительная валидация:**
Агент покажет:
- Путь до папки с тест-кейсами
- Таблицу тест-кейсов (Test ID, Название, Приоритет, Техника)
- Параметры импорта

**После импорта:** Агент обновит файл с тест-кейсами, заменив временные ID на полученные из TMS.

**Валидация:** Два запроса подтверждения - перед запуском и после обновления ID.

### Этап 5: Ручное тестирование и формирование отчёта

**Параметры выполнения:**
- **Base URL:** `https://sc-vp.ift.pd04.pvw.sbt/web/v2/`
- **Authorization:** Bearer токен (должен быть актуальным)
- **Browser:** Chrome / Firefox / Safari

**Требования к выполнению:**
1. ✅ Использовать только данные из Этапа 1 (test-model)
2. ✅ Выполнить ВСЕ тест-кейсы из Этапа 2
3. ✅ **НЕ изменять** ожидаемые результаты из тест-кейсов
4. ✅ Фиксировать фактические результаты для каждого шага
5. ✅ Делать скриншоты при дефектах
6. ✅ Указывать версию UI (web1 или web2)

## Критические правила

1. 🛑 **НИКОГДА** не переходить к следующему этапу без явного подтверждения пользователя
2. 🛑 **НИКОГДА** не изменять ожидаемые результаты тестов из Этапа 2
3. 🛑 **ВСЕГДА** использовать актуальный Bearer токен для Этапа 5
4. 🛑 **ВСЕГДА** использовать только UI-узлы из test-model (не API)
5. ✅ **ВСЕГДА** запрашивать валидацию в явном виде после каждого этапа
6. ✅ **ВСЕГДА** делать скриншоты при дефектах
7. ✅ **ВСЕГДА** указывать версию UI (web1 или web2)
8. ✅ **ВСЕГДА** использовать паттерны из `patterns.md` для шагов и результатов

## Устранение неполадок

### Ошибка авторизации в TMS
- Проверьте учётные данные в `config.md`
- Убедитесь, что срок действия пароля не истёк

### Ошибка Bearer токена (Этап 5)
- Запросите новый токен через API аутентификации
- Обновите токен в параметрах выполнения

### Скрипт импорта не найден
- Убедитесь, что `import_tests_to_tms_full.py` находится в рабочей директории
- Проверьте права на выполнение скрипта

### Ошибка получения UI-узлов
- Проверьте доступность vcs-knowledge
- Убедитесь, что test-model содержит нужный узел
