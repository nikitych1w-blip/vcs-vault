## Этап 1: Генерация тест-кейсов

### Задача
Создать тест-кейсы на основе OpenAPI спецификации, используя различные техники тест-дизайна.

### Входные данные
- **Спецификация:** `[ВСТАВИТЬ СПЕКУ]` (например: `swagger web v2.md` или `openapi_http_v3_bundle.yaml`)
- **Метод:** `[ВСТАВИТЬ МЕТОД]` (например: `POST /members`)

### Техники тест-дизайна (обязательно к использованию)
1. ✅ Эквивалентное разделение (Equivalence Partitioning)
2. ✅ Граничные значения (Boundary Value Analysis)
3. ✅ Попарное тестирование (Pairwise Testing)
4. ✅ Предугадывание ошибок (Error Guessing)
5. ✅ Тестирование на основе состояний (State-Based Testing)
6. ❌ Исключить: проверки на инъекции (SQL, CSS, XSS и т.п.)

### Формат вывода
- Использовать строго шаблон из файла `shablon.md`
- Имя файла: `[METHOD]_[ENDPOINT].md` (пример: `POST_MEMBERS.md`)
- Структура тест-кейса:
  ```markdown
  ## {test_id}
  **Id:** {test_id}
  **Название:** {название}
  **Приоритет:** {critical/major/minor}
  **Уровень теста:** {api/ui/integration}
  **Теги:** {теги с указанием техники тест-дизайна}
  **Шаги:**
  **Шаг N:** {описание}
  **Тестовые данные:** {значение}
  **Ожидаемый результат:** {ожидаемый результат}
  ```

### Требования к тест-кейсам (критично)
1. **Один код ответа на тест-кейс:**
   - В ожидаемом результате шага с HTTP-запросом указывать **СТРОГО ОДИН** код ответа (200, 400, 401, 403, 404 и т.д.)
   - Запрещено указывать диапазоны или множественные коды (например, "200-299" или "200/201")
   - Пример корректного ожидания: `"Ожидаемый результат: Запрос выполнен. Статус ответа 200 OK"`
   - Все кейсы должны содержать тег `agent-generated`

2. **Проверка тела ответа по схеме спецификации:**
   - В ожидаемом результате указывать проверку структуры тела ответа согласно схеме из OpenAPI спецификации
   - Для успешных ответов (2xx): проверять наличие обязательных полей из схемы
   - Для ошибок (4xx, 5xx): проверять стандартную структуру ошибок API
   - Пример корректного ожидания: `"Ожидаемый результат: Тело содержит объект с полями: id (integer), name (string), description (string) согласно схеме Response"`

3. **Формат тестовых данных:**
   - Для тела ответа в ожидаемом результате использовать компактное представление JSON
   - Указывать только ключевые поля для проверки, не обязательно полное тело
   - Пример: `{"pagination": {"total_items": 0}, "data": []}`

4. **Требование к тегам:**
   - Все кейсы должны содержать тег `agent-generated`
   - Общее количество тегов не больше 6

### ⚠️ ТОЧКА ВАЛИДАЦИИ #1
После генерации файла:
1. Показать пользователю путь к созданному файлу
2. Показать краткую сводку: количество тест-кейсов, распределение по техникам
3. **Запросить подтверждение:** "Проверьте файл [имя_файла]. Все ли тест-кейсы корректны? Продолжить?"
4. Не продолжать до получения подтверждения

## Этап 2: Анализ структуры папок в TMS

### Задача
Определить место для создания папки с тест-кейсами в системе управления тестированием.

### Действия
1. Выполнить POST запрос:
   - **URL:** `https://portal.works.prod.sbt/swtr/extension/plugin/v2/rest/api/swtr_tms_plugin/v1/folder/root`
   - **Method:** POST
   - **Body:**
     ```json
     {
       "type": "TEST_CASE",
       "spaceId": {
         "code": "VCS"
       }
     }
     ```
   - **Authorization:** Basic Auth (данные из `config.md`)

2. Определить путь для папки в зависимости от источника спецификации:
   | Исходный файл | Путь в TMS |
   |--------------|------------|
   | `openapi_ui_bundle.yaml` | `SC v2 → 2. api web → 4. v2 → [новая папка]` |
   | `openapi_http_v3_bundle.yaml` | `C v2 → 1. API → 2. api v3 → [новая папка]` |

3. Проверить существование папки:
   - Если **существует**: запомнить `folder_id` (UUID), пропустить создание
   - Если **не существует**: запомнить `parent_folder_id` для создания

### ⚠️ ТОЧКА ВАЛИДАЦИИ #2
После анализа:
1. Показать пользователю:
   - Найденный/предполагаемый путь в TMS
   - ID папки (если существует) или ID родительской папки
2. **Запросить подтверждение:** "Путь для тест-кейсов: [путь]. ID папки: [id]. Продолжить?"
3. Не продолжать до получения подтверждения

## Этап 3: Импорт тест-кейсов в TMS

### Задача
Импортировать созданные тест-кейсы в систему управления тестированием.

### Параметры для скрипта `import_tests_to_tms_full.py`
| Аргумент | Значение | Источник |
|----------|----------|----------|
| `--create-cycle` | флаг создания цикла | - |
| `--username` | логин пользователя | `config.md` |
| `--password` | пароль пользователя | `config.md` |
| `--folder` | UUID папки тест-кейсов | Этап 2 (если папка существует) |
| `--folder-name` | Название папки | Формат: "[METHOD] [endpoint]" (пример: "PATCH members") |
| `--parent-folder` | ID родительской папки | Этап 2 (если папка не существует) |
| `--cycle-name` | Название цикла тестов | Аналогично folder-name |
| `[путь к файлу]` | Путь к файлу с тест-кейсами | Этап 1 |

**1. Путь до папки с тест-кейсами:**


**2. Сводная таблица тест-кейсов:**
| Test ID | Название | Приоритет | Техника |
|---------|----------|-----------|---------|
| ... | ... | ... | ... |

**3. Параметры импорта:**
```
Username: [значение]
Folder: [значение или "будет создано"]
Folder Name: [значение]
Cycle Name: [значение]
```

### ⚠️ ТОЧКА ВАЛИДАЦИИ #3

2. Обновить файл из Этапа 1: заменить временные **Id:** на полученные из TMS

## Этап 5: Прогон тестов и формирование отчета

### Задача
Выполнить все тест-кейсы из Этапа 1 на сгенерированных данных и создать отчет.

### Параметры выполнения
| Параметр | Значение |
|----------|----------|
| **Base URL** | `https://sc-vp.ift.pd04.pvw.sbt/web/v2/` |
| **Authorization** | Bearer токен (ВСЕГДА АКТУАЛЬНЫЙ) |
| **Токен** | `[ВСТАВИТЬ ТОКЕН]` |

### Требования к выполнению
1. ✅ Использовать **ТОЛЬКО** данные, сгенерированные на Этапе 4
2. ✅ Выполнить **ВСЕ** тест-кейсы из файла Этапа 1
3. ✅ **НЕ изменять** ожидаемые результаты из тест-кейсов
4. ✅ Фиксировать фактические результаты для каждого шага

### Формат отчета
Создать файл по шаблону `test_results_shablon.json`:

```json
{
  "summary": {
    "total": <общее количество тестов>,
    "passed": <количество успешных>,
    "failed": <количество неудачных>,
    "success_rate": "<процент успеха>"
  },
  "results": [
    {
      "test_id": "<ID теста>",
      "name": "<Название теста>",
      "passed": <true/false>,
      "expected_status": <ожидаемый HTTP статус>,
      "actual_status": <фактический HTTP статус>,
      "response": "<тело ответа>"
    }
  ]
}
```

### ⚠️ ТОЧКА ВАЛИДАЦИИ #6 (ФИНАЛЬНАЯ)
После выполнения всех тестов:
1. Показать сводку результатов:
   ```
   Всего тестов: X
   Успешно: Y
   Не успешно: Z
   Процент успеха: W%
   ```
2. Показать путь к файлу с результатами
3. **Запросить подтверждение:** "Тестирование завершено. Результаты сохранены в [файл]. Задача выполнена."
