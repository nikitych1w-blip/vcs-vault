## ADDED Requirements

### REQ-RS-001: Поиск репозиториев по строке и области проекта
**Узел**: API / web2 / Проект / Репозитории / Поиск
**Уровень**: lv6
**Теги**: #SC #api #web2 #repo #project #ПАО
**Версия API**: web2
**Компонент**: API

Система SHALL выполнять поиск репозиториев через `GET /repos/search` по параметру `q` в области доступных пользователю проектов, а при передаче `project_name` ограничивать поиск указанным проектом.

#### Scenario: Поиск по всем доступным проектам
- **WHEN** аутентифицированный пользователь вызывает `GET /repos/search` с параметром `q` и без `project_name`
- **THEN** система возвращает результаты, совпадающие с `q`, только из доступных пользователю проектов

#### Scenario: Поиск в рамках проекта
- **WHEN** аутентифицированный пользователь вызывает `GET /repos/search` с параметрами `q` и `project_name`
- **THEN** система возвращает результаты поиска только внутри указанного проекта

### REQ-RS-002: Фильтрация результатов поиска
**Узел**: API / web2 / Проект / Репозитории / Поиск / Фильтры
**Уровень**: lv7
**Теги**: #SC #api #web2 #repo #ПАО
**Версия API**: web2
**Компонент**: API

Система SHALL поддерживать фильтрацию результатов поиска по параметрам `language`, `archived`, `is_watching`, `is_favorite`, `is_in_source_hub`, где булевы фильтры работают в tri-state режиме (`true`, `false`, не задано), а `language` принимает список языков через запятую без пробелов.

#### Scenario: Применён фильтр по языкам
- **WHEN** пользователь вызывает `GET /repos/search` с параметром `language=typescript,python`
- **THEN** в выдачу попадают только репозитории, у которых язык соответствует переданным значениям

#### Scenario: Булев фильтр не передан
- **WHEN** пользователь вызывает `GET /repos/search` без одного из булевых фильтров (`archived`, `is_watching`, `is_favorite`, `is_in_source_hub`)
- **THEN** система не ограничивает выдачу по этому признаку и возвращает оба типа значений

### REQ-RS-003: Сортировка и пагинация
**Узел**: API / web2 / Проект / Репозитории / Поиск / Сортировка и пагинация
**Уровень**: lv7
**Теги**: #SC #api #web2 #repo #ПАО
**Версия API**: web2
**Компонент**: API

Система SHALL поддерживать сортировку результатов через параметр `sort` (поля `update`, `name`, `stars`, `forks`, направление `asc/desc`) и пагинацию через `page` и `limit`, возвращая согласованные метаданные пагинации.

#### Scenario: Запрос с сортировкой и пагинацией
- **WHEN** пользователь вызывает `GET /repos/search` с параметрами `sort=name:asc,stars:desc`, `page=2`, `limit=10`
- **THEN** система возвращает вторую страницу результатов, отсортированных согласно переданному порядку

#### Scenario: Сортировка не задана
- **WHEN** пользователь вызывает `GET /repos/search` без параметра `sort`
- **THEN** система применяет поведение сортировки по умолчанию, определённое backend-логикой для endpoint

### REQ-RS-004: Формат успешного ответа списка репозиториев
**Узел**: API / web2 / Проект / Репозитории / Поиск / Ответ
**Уровень**: lv8
**Теги**: #SC #api #web2 #repo #ПАО
**Версия API**: web2
**Компонент**: API

Система SHALL при успешном запросе возвращать `200` с телом `ReposShortList`, содержащим обязательные блоки `pagination` и `data[]`, где каждый элемент `data` соответствует контракту `RepoShortListItem`.

#### Scenario: Успешная выдача с непустым списком
- **WHEN** найден хотя бы один репозиторий по условиям запроса
- **THEN** система возвращает `200` и `data[]` с элементами `RepoShortListItem`, включая обязательные поля идентификации, видимости, активности и временных меток

#### Scenario: Успешная выдача без совпадений
- **WHEN** по условиям запроса совпадения не найдены
- **THEN** система возвращает `200`, пустой `data[]` и корректный объект `pagination`

### REQ-RS-005: Обработка ошибок и невалидных параметров поиска
**Узел**: API / web2 / Проект / Репозитории / Поиск / Ошибки
**Уровень**: lv8
**Теги**: #SC #api #web2 #repo #ПАО
**Версия API**: web2
**Компонент**: API

Система SHALL возвращать ошибки `400/401/403/404/500` согласно контракту endpoint, включая `400` для невалидных параметров запроса (например, некорректный формат `sort`).

#### Scenario: Невалидный параметр сортировки
- **WHEN** пользователь передаёт `sort`, не соответствующий поддерживаемому формату
- **THEN** система возвращает `400 Bad Request` с описанием ошибки валидации

#### Scenario: Запрос без аутентификации
- **WHEN** вызов `GET /repos/search` выполнен без валидной аутентификации
- **THEN** система возвращает `401 Unauthorized`

#### Scenario: Нет прав на область поиска
- **WHEN** пользователь аутентифицирован, но не имеет прав на запрошенную область данных
- **THEN** система возвращает `403 Forbidden`

#### Scenario: Проект не найден
- **WHEN** в запросе передан `project_name`, который отсутствует или недоступен
- **THEN** система возвращает `404 Not Found`

#### Scenario: Внутренняя ошибка обработки
- **WHEN** при выполнении поиска происходит непредвиденная ошибка backend-компонентов
- **THEN** система возвращает `500 Server Error`
