## Context

Изменение `vcs-00000` добавляет в `web2` единый пользовательский сценарий поиска и отбора репозиториев через `GET /repos/search`.
FE-артефакт фиксирует UI/UX-решения для экрана списка репозиториев: состав компонентов, состояния, интеграцию с API-контрактом и поведение навигации.

Источники:
- `openspec/changes/vcs-00000/sa/proposal.md`
- `openspec/changes/vcs-00000/sa/specs/repos-search/spec.md`
- `openspec/changes/vcs-00000/be/design.md`
- `sources/vcs-api/api/openapi/ui/domains/repos/{routes,params,schemas,responses}.yaml`
- `макет`: `/home/khalbashkeev.p.s@sbertech.ru/Development/vcs-vault/макет.png`

Ограничения FE-стека:
- `web2`, React 18 + FSD (`app -> pages -> widgets -> features -> entities -> shared`);
- API-запросы только через Orval-генерацию (`shared/api/generated`), без ручного `useQuery`;
- базовый API-префикс в приложении: `/apifront/web/v2` (`window.__CONTEXT_API_PATH__`).

## Goals / Non-Goals

**Goals:**
- Спроектировать экран поиска репозиториев по макету: поиск, быстрые фильтры, сортировка, карточки, пагинация.
- Зафиксировать маппинг query-параметров UI в контракт `GET /repos/search` (`q`, tri-state фильтры, `language`, `sort`, `page`, `limit`, `project_name`).
- Зафиксировать обязательные UI-состояния: `loading`, `success`, `empty`, `error/{код}`.
- Описать набор полей `RepoShortListItem`/`Pagination`, которые отображаются в UI.
- Согласовать навигацию и deep-linking через URL query params.

**Non-Goals:**
- Изменение OpenAPI-контракта (`sources/vcs-api`) или backend-алгоритмов поиска.
- Ручные правки `openapi_ui_bundle.yaml` и `shared/api/generated/**`.
- Добавление нового домена API, side effects (Kafka/Git) или миграций БД.
- Редизайн вне границ предоставленного макета и требований SA.

## Decisions

### Макет

Источник: `proposal.md` -> `Макет` -> `/home/khalbashkeev.p.s@sbertech.ru/Development/vcs-vault/макет.png`.

Ключевые элементы экрана, которые принимаются в реализацию:
- верхняя панель фильтрации: строка поиска + чипы быстрых фильтров (`Избранные`, `Наблюдаю`, `InSourceHub`) + фильтр архивности;
- селектор сортировки справа (`Недавно обновлённые` как основной сценарий `sort=update:desc`);
- список карточек репозиториев с названием, бейджами типа (`Приватный`, `InSourceHub`, `Архивный`), описанием, топиками и временем обновления;
- нижняя пагинация (`предыдущая/следующая`, переход по страницам).

Детализация пиксель-уровня и точных отступов не фиксируется в артефакте: реализация на компонентах дизайн-системы `@sds-eng/base`.

### Компоненты

| Слой FSD | Компонент | Тип изменения | Назначение |
|---|---|---|---|
| `pages` | `ProjectRepositoriesPage` (или текущая страница раздела репозиториев) | modified | Оркестрация query-state, вызов Orval-хука, связывание блоков экрана |
| `widgets` | `RepositoriesSearchToolbar` | new/modified | Поисковая строка, быстрые фильтры, фильтр архивности/языков, селектор сортировки |
| `widgets` | `RepositoriesList` | modified | Рендер списка карточек, переключение состояний `loading/empty/error/content` |
| `entities/repository` | `RepositoryCard` | modified | Отрисовка полей `RepoShortListItem`, переход в репозиторий, `data-testid` на ключевых действиях |
| `features` | `RepositoriesPagination` | new/modified | Управление `page/limit`, синхронизация с URL и `pagination` из API |

Принципы реализации:
- импорт между срезами только через публичный API (`index.ts`);
- ключевые интерактивные элементы покрываются `data-testid` (по styleguide);
- параметры пути читаются через `useCustomParams()` с дефолтами, query — через `useQueryParams()`.

### Состояния

| Состояние | Условие | Отображение в UI | Действие пользователя |
|---|---|---|---|
| `loading` | первый вход или смена фильтра/страницы/сортировки | `Spinner` в контентной зоне, тулбар остаётся доступным | ждать завершения, можно менять фильтры |
| `success` | `200`, `data.length > 0` | список карточек + пагинация | открыть репозиторий, менять фильтры/сортировку |
| `empty` | `200`, `data.length === 0` | `PageError type="EMPTY"` с текстом "Ничего не найдено" | очистить фильтры или изменить запрос |
| `error/400` | невалидные query-параметры (`sort/language/page/limit`) | error-block с понятным текстом валидации (`detail/validation`) | сбросить некорректные фильтры в дефолт |
| `error/401` | нет валидной cookie-аутентификации | UI-экран не показывается, переход в auth-flow хоста | повторная аутентификация |
| `error/403` | нет прав на область поиска | no-access состояние (`PageError`, без утечки деталей) | вернуться к доступному разделу |
| `error/404` | `project_name` не найден | not-found состояние (пустой контекст проекта) | перейти в другой проект/убрать `project_name` |
| `error/500` | внутренняя ошибка сервиса | `PageError type="SERVER_ERROR"` + кнопка "Повторить" | повторный запрос с текущими параметрами |

### API-интеграция

| Эндпоинт | Версия | Когда вызывается |
|---|---|---|
| `GET /repos/search` | `web2` (`/apifront/web/v2/repos/search`) | initial load, изменение `q`, фильтров/сортировки, смена страницы/лимита, retry после ошибок |

Техническое решение:
- источник хука — Orval из `operationId: GetReposSearch` (ожидаемый хук: `useGetReposSearch`);
- параметры запроса формируются из URL query state;
- сериализация `language` только CSV без пробелов (`typescript,python`);
- tri-state фильтры (`archived`, `is_favorite`, `is_watching`, `is_in_source_hub`) не отправляются, если выбран режим "все";
- при изменении `q`/фильтра/сортировки `page` сбрасывается в `1`.

### Данные для отображения

| UI-элемент | Поле API | Примечание |
|---|---|---|
| Аватар репозитория | `data[].avatar_url` | fallback-иконка при пустом/битом URL |
| Название карточки | `data[].name` | основной заголовок |
| Полный путь | `data[].full_name` | вторичная строка/tooltip |
| Бейджи типа | `data[].private`, `data[].internal`, `data[].template`, `data[].fork`, `data[].archived`, `data[].is_in_source_hub` | чипы состояния репозитория |
| Описание | `data[].description` | пустое значение -> заглушка "Описание не заполнено" |
| Топики | `data[].topics[]` | рендер ограниченного числа чипов + overflow |
| Признак "Избранное" | `data[].is_starring` | состояние иконки/чипа в карточке |
| Признак "Наблюдаю" | `data[].is_watching` | состояние иконки в карточке |
| Время обновления | `data[].updated_at` | формат "обновлено N ... назад" |
| Счётчики | `data[].stars_count`, `data[].watchers_count`, `data[].forks_count` | блок метрик карточки |

### Навигация

- В рамках `vcs-00000` новый роут не добавляется: дорабатывается существующий экран раздела `UI / Проект / Репозитории`.
- Path params берутся из текущего роута проекта (`project_name`/алиас в FE-роутере) и используются для формирования `project_name` в API-query при scoped-поиске.
- Query params экрана (источник истины для состояния):
  - `q`
  - `is_favorite`
  - `is_watching`
  - `is_in_source_hub`
  - `language`
  - `archived`
  - `sort`
  - `page`
  - `limit`
- Правила переходов:
  - ввод в поиск/смена фильтра/смена сортировки -> обновление query + сброс `page=1`;
  - переход по странице -> обновление `page` (и при необходимости `limit`);
  - клик по карточке -> переход на существующую страницу репозитория;
  - URL должен быть shareable (перезагрузка сохраняет выбранные фильтры).
- Если в будущем появится отдельный роут под эту страницу, обязательно синхронизировать regex в `web_src/js/bridge/gitea-react-adapter.js`.

## Risks / Trade-offs

- Есть риск частых `400`, если UI сериализует `sort/language/page/limit` не по контракту; снижается централизованным query-serializer.
- Tri-state фильтры усложняют синхронизацию UI/URL/API (важно не отправлять `false` вместо "не задано" там, где требуется оба состояния).
- Поиск по большому набору доступных проектов может давать заметные задержки; в UI нужен debounce для `q` и сохранение предыдущих данных при перезапросе.
- Разделение `403` и `404` для `project_name` должно быть визуально различимо, иначе пользователю сложно понять причину отсутствия результатов.
- Макет фиксирует ключевые элементы, но не задаёт все микро-состояния; часть UX-решений (тексты, заглушки, responsive-breakpoints) остаётся в зоне FE + дизайн-системы.
