## Контекст задачи

Ты создаёшь тест-план на основе SA-требований.

При работе над каждым артефактом:

- Читай `sa/proposal.md` — источник намерения
- Читай `sa/specs/` — требования от SA: это основа для тест-кейсов
- Сверяй узлы модели с SA-артефактами до написания TC
- Все артефакты размещай в `changes/{change-name}/qa/`

---

## Инструкции по артефактам

### qa/test-plan.md

```markdown
## Test Plan: <название изменения>

### Затрагиваемые узлы
| Путь | lv | Теги | Текущий статус |
|------|----|------|----------------|

### Позитивные сценарии
- [ ] TC: <краткое описание> — узел lv<N>

### Негативные сценарии
- [ ] TC: <краткое описание> — узел lv<N>

### Граничные случаи
- [ ] TC: <краткое описание> — узел lv<N>

### Приоритетные кейсы (#ПАО)
- [ ] TC: <краткое описание> — узел lv<N>, #ПАО

### Gate релиза
- [ ] Все #ПАО-узлы имеют status: covered
- [ ] SA подтвердил список изменённых узлов
- [ ] Метрика покрытия рассчитана
```

### qa/tc/{node}.md

```markdown
---
id: TC-XXXX
node: <путь в test-model>
lv: <N>
tags: [...]
api_version: <web2 / v1 / v2 / ...>
status: covered
priority: high / medium / low
---

## TC-XXXX: <название>

**Узел**: `<путь>` (lv<N>)
**Теги**: `#...`

### Предусловия
### Шаги
### Ожидаемый результат
### Альтернативные сценарии
### Связанные узлы
```

Правила:
- Координата узла обязательна в каждом TC
- lv7–lv8 — основной уровень детализации
- #ПАО-узлы покрывай первыми
- Разное поведение в разных версиях API — разные TC
- Deprecated-узлы не тестируй, деактивируй существующие TC
