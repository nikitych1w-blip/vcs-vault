## Контекст задачи

Ты создаёшь артефакты OpenSpec для изменения, описанного в `sa/proposal.md`.

При работе над каждым артефактом:

- Читай `sa/proposal.md` — это источник намерения
- Читай `sa/specs/` — это действующие требования (если есть)
- Все новые артефакты размещай в `changes/{change-name}/sa/`

---

## Инструкции по артефактам

### proposal.md

Опиши бизнес-намерение изменения:

- Какую проблему решаем
- Для кого
- Что выходит за scope
- Предлагаемый подход

Не описывай реализацию — только намерение и контекст.

### sa/specs/{node}/spec.md

Каждый файл — один узел test-model.

Используй формат:

```markdown
## ADDED / MODIFIED / REMOVED

### REQ-XXX: <название>

**Узел**: <путь в test-model>
**Уровень**: lv<N>
**Теги**: #... #...
**Версия API**: <v1 / v2 / v3 / web1 / web2>
**Компонент**: <UI / API / CLI>

#### Предусловия
#### Основной сценарий
#### Альтернативные сценарии
#### Логика реализации
#### Ограничения / Edge cases
#### Связанные эндпоинты / элементы UI
```

Правила:
- Координата узла обязательна в каждом требовании
- Указывай версию API явно
- Разное поведение в разных версиях — разные требования
- Deprecated-узлы (`(удалить раздел)`) не анализируй
- Описывай логику изнутри: валидации, трансформации, взаимодействие сервисов

### API-контракт

Если изменение затрагивает API, добавь контракт в `sa/specs/{node}/api.md`:

```markdown
## <METHOD> /api/<версия>/<endpoint>

**Узел**: <путь в test-model>
**Версия**: #v1 / #v2 / #v3 / #web1 / #web2
**Аутентификация**: access-token / JWT / —

### Параметры запроса
### Тело запроса
### Успешный ответ (2xx)
### Коды ошибок
```
