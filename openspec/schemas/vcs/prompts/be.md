## Контекст задачи

Ты создаёшь технические артефакты OpenSpec на основе SA-требований.

При работе над каждым артефактом:

- Читай `sa/proposal.md` — источник намерения
- Читай `sa/specs/` — требования и API-контракты от SA
- Все артефакты размещай в `changes/{change-name}/be/`

---

## Инструкции по артефактам

### be/design.md

```markdown
## BE Design: <название изменения>

### Endpoint(s)
<METHOD> <path> — краткое описание

### Auth
Механизм аутентификации и авторизации

### Flow
1. Валидация входных параметров
2. Проверка прав доступа
3. Бизнес-логика
4. Взаимодействие с БД / внешними сервисами
5. Формирование ответа

### Error Handling
| Условие | Код | Сообщение |
|---------|-----|-----------|

### DTO / Модели данных
Описание структур данных запроса и ответа

### Зависимости
Внешние сервисы, интеграции (Kafka, Git, LDAP и др.)

### Архитектурные решения
Нестандартные решения с обоснованием
```

Правила:
- Каждое решение трассируемо к требованию из `sa/specs/`
- Указывай версию API явно (v1 / v2 / v3 / web1 / web2)
- Kafka-события: описывай топик, схему сообщения, триггер
- Deprecated-функциональность не реализуй

### be/tasks.md

```markdown
## BE Tasks: <название изменения>

### Реализация
- [ ] Роутинг: зарегистрировать эндпоинт
- [ ] Handler: валидация параметров
- [ ] Service: бизнес-логика
- [ ] DTO: описать структуры
- [ ] Маппинг: внешний ответ → internal DTO

### Интеграции
- [ ] <интеграция, если есть>

### Тесты
- [ ] Unit: handler — позитивный кейс
- [ ] Unit: handler — негативные кейсы (400, 401, 403, 404)
- [ ] Unit: service — бизнес-логика

### Документация
- [ ] OpenAPI-спека обновлена
- [ ] Swagger-документация сгенерирована
```
