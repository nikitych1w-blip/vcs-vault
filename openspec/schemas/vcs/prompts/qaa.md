## Контекст задачи

Ты создаёшь план автоматизации на основе ручных тест-кейсов QA.

При работе над каждым артефактом:

- Читай `sa/proposal.md` — источник намерения
- Читай `sa/specs/` — требования и API-контракты
- Читай `qa/test-plan.md` и `qa/tc/` — ручные TC для автоматизации
- Читай `be/design.md` — структуры запросов и ответов
- Все артефакты размещай в `changes/{change-name}/qaa/`

---

## Инструкции по артефактам

### qaa/tasks.md

```markdown
## QAA Tasks: <название изменения>

### E2E тесты (UI)
Покрывают узлы #UI и #FE:
- [ ] e2e: <сценарий> — узел lv<N>, TC-XXXX

### Integration тесты (API)
Покрывают узлы #api, #v1, #v2, #web2:
- [ ] integration: <эндпоинт> — позитивный кейс
- [ ] integration: <эндпоинт> — негативные кейсы (400/401/403/404)
- [ ] integration: маппинг полей DTO

### Contract тесты
- [ ] contract: ответ соответствует OpenAPI-спеке
- [ ] contract: схема Kafka-события (если применимо)

### Приоритетные (#ПАО)
- [ ] <тест> — узел lv<N>, #ПАО

### Инфраструктура
- [ ] Тестовые данные (фикстуры / моки) подготовлены
- [ ] CI: тесты добавлены в pipeline
```

Правила:
- Каждый авто-тест трассируется к TC из `qa/tc/` через `TC-XXXX`
- Координата узла обязательна
- Разные версии API — разные тесты
- #ПАО-узлы автоматизируй первыми
- Deprecated-функциональность не покрывай
