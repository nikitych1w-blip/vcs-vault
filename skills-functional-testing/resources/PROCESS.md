## Требования к регрессионной модели

**VCS-14135**

Вопрос формирования тестового регрессионного набора всегда стоит перед выпуском релиза.

### Требования:
1. Тестовый цикл регрессионного набора должен состоять из **критичных и блокирующих** тестовых кейсов
2. Актуализация модели перед каждым релизом
3. Сбор всех связанных тест-кейсов по релизу

## Как сформировать регрессионный цикл

**VCS-17200**

### Подготовка
- Тестовый цикл должен состоять из критичных и блокирующих тестовых кейсов

### Пошаговая инструкция
1. Перейти в WORKS → Циклы
2. Нажать «+ Создать тестовый цикл»
3. Название: «Регресс релиза X.X.X»
4. Добавить тест-кейсы через TQL:
   ```tql
   space = "VCS" 
   AND unit in("VCS-7764", "VCS-8168", ... , "VCS-8833", "VCS-8797")
   AND priority in("critical", "blocker")
   ```
5. Или добавить актуальные тест-кейсы отмеченные метками "Критический" и "Блокирующий"
   ```tql
   space = "VCS" 
   AND test_case_status in("relevant") 
   AND priority in("critical", "blocker") 
   AND automated !~ "yes" 
   AND summary !~ "негатив" AND summary !~ "отрицат"
   ```

## Как сформировать чистового тестирования (НФ) по релизу

**VCS-17201**

### Подготовка
- Тестовый цикл чистового набора должен состоять из задач релиза

### Пошаговая инструкция
1. Собрать тестовые циклы по релизу через TQL:
   ```tql
   space = "VCS" AND release_name ~ "X.X.X"
   ```

2. Сформировать TQL для всех тест-кейсов релиза:
   ```tql
   space = "VCS" 
   AND suit = "test_case" 
   AND unit IN linkedUnitsOf("unit IN ('VCS-X.X.X', 'VCS-X.X.1')", "Тест-кейс в тест-цикле")
   ```

3. Создать тестовый цикл: «НФ Релиза X.X.X»

4. Добавить тест-кейсы через TQL (п.2)

5. Использовать Postman коллекцию для массового назначения:
   - Скачать коллекцию и окружение
   - Заполнить Variables (cookies, testCycleId, assigned_to, unit IN)
   - Выполнить runner для распределения тест-кейсов

## Определение приоритета тестового сценария

**VCS-14828**

### Уровни приоритета
| Уровень | Описание |
|---------|----------|
| **Blocker** | Проверка базовых функций, без которых продукт невозможно использовать |
| **Critical** | Важнейшие функциональные тесты, покрывают ключевые сценарии |
| **Major** | Средней важности кейсы, проверяющие менее частные сценарии |
| **Minor** | Менее важные кейсы, редко используемые функции |
| **Trivial** | Низкоприоритетные кейсы, мелкие детали, редкие случаи |

### Базовые вопросы для определения:
1. Это единственный успешный путь фичи? → **Blocker**
2. Поломка приведет к потере данных или краже доступа? → **Critical**
3. Сломан ли основной механизм работы с данными? → **Critical**
4. Это мешает пользоваться функцией, но есть другой способ? → **Major**
5. Это касается только внешнего вида или удобства? → **Minor**

| Уровень | Критерий | Описание |
|---------|----------|----------|
| **Blocker** | Главный - успешный путь | Самая важная команда, без неё функция бесполезна |

## Правила определения уровня приоритета тест-кейса

**VCS-9744**

### Принципы:
- Чем важнее проверяемый сценарий для пользователя/бизнеса — тем выше приоритет
- Чем чаще используется функция — тем выше её значимость
- Более высокие приоритеты — основной функционал и бизнес-процессы

### Примеры:
- **Blocker**: авторизация, создание репозитория, базовая навигация
- **Critical**: создание pull-request, добавление участников, назначение прав
- **Major**: фильтрация поиска, экспорт данных, интеграции
- **Minor**: поведение при большом количестве сессий, нестандартные форматы
- **Trivial**: аватары, предупреждающие сообщения

## Чек-лист must have API проверок

**VCS-14081**

### Критические сценарии:
1. **401 — аутентификация**
   - Отсутствие заголовка Authorization
   - Пустое значение Authorization
   - Невалидный токен
   - Отозванный токен
   - Несуществующий BasicAuth
   - Невалидный пароль BasicAuth

2. **403 — права доступа**
   - Доступ к приватному без выдачи прав
   - Проверка по ролевой модели
   - Скоуп токена (если указан в СТ)

### Важные сценарии:
- Валидация текста: латиница, кириллица, спецсимволы, эмодзи
- Длина текста (учесть байты на символ для кириллицы)
- Аудит изменений (создание, изменение, удаление)
- Корректное отображение в Swagger

## Чек лист для ревью тест кейсов

**VCS-11033**

### 1. Структура и оформление
- [ ] Название: краткое, отражает суть (без "Успешный", "Позитивный")
- [ ] Приоритет: указан и соответствует важности
- [ ] Предусловия: корректно описаны
- [ ] Постусловия: возврат в исходное состояние
- [ ] Связь с требованиями: ссылка на СТ/задачи

### 2. Шаги выполнения
- [ ] Логичная последовательность, пронумерованы
- [ ] Один шаг — одно действие
- [ ] Конкретность: локаторы, названия кнопок, данные
- [ ] ОР: для каждого шага или группы
- [ ] Полнота: ОР описывает изменения на UI и состояние системы

### 3. Содержание и покрытие
- [ ] Понятна цель теста
- [ ] Один сценарий на кейс
- [ ] Четко определен тип (позитивный/негативный)
- [ ] Тестовые данные: явно указаны или есть ссылка
- [ ] Валидация ошибок: конкретное сообщение
- [ ] Отсутствие избыточности

### 4. Техническая корректность
- [ ] Соответствует текущей версии функциональности
- [ ] Автоматизуемость: без субъективных проверок
- [ ] Язык и стиль: повелительный, без ошибок
- [ ] Независимость: выполним отдельно

### 5. Критические недостатки
- ❌ Шаги "и т.д." — признак неполноты
- ❌ "Все работает корректно" — неконкретно
- ❌ Требует знаний "как в старом кейсе"
- ❌ Слишком длинный (более 15-20 шагов)

## Как определить какие кейсы взять в регресс

**VCS-14380**

### Для итерации функциональности (задача/дефект):

1. **Связанный функционал** — только критические приоритетные кейсы
   - Задай себе вопрос "А что если?"
   - Ключевой функционал: создание, изменение, просмотр, права доступа

2. **Исходящий функционал** — критические тесты без которых приложение нецелесообразно
   - Обычно 1-2 теста (авторизация, ролевая модель)

3. **Smoke и чек-лист** для задачи
   - Smoke: user flow, цепочка событий
   - Чек-лист: конкретный тест-план проверок

## Как определить какие тесты берём в регресс релиза

**VCS-14390**

1. Проверить таблицу API v3 покрытия (VCS-10691)
   - Не покрыто АТ → ручной регресс (только критичный функционал)

2. Проверить OpenAPI bundle
   - Не покрыто АТ → ручной регресс (важные части интеграции + безопасность)
   - Негативные кейсы и тестирование на отказ добавлять только если блокирующие

## Правила написания тест-кейса

**VCS-7000**

### Этапы создания:

**Этап 0: Предусловия**
- Требования (СТ, User Stories, макеты, техническая документация)
- Чеклист покрытия AC

**Этап 1: Подготовка окружения**
- Собрать ветку и задеплоить
- Провести smoke-тест

**Этап 2: Анализ и проектирование**
- Определение приоритетов (Blocker/Critical/Major/Minor/Trivial)
- Выбор техник тест-дизайна (эквивалентное разделение, граничные значения и т.д.)

**Этап 3: Создание тест-кейса**
- Название: краткое и понятное
- Описание: цель, контекст
- Предусловия: что должно быть выполнено
- Шаги: последовательность действий

**Этап 4: Ревью и доработка**
- Пиринг-ревью другими тестировщиками
- Внесение правок

**Этап 5: Финализация**
- Утверждение и перевод в статус «Актуальный»

### Структура тест-кейса:
1. **Название** — краткое описание сути проверки
2. **Описание** — подробное описание цели теста
3. **Предусловия** — что должно быть выполнено до начала
4. **Шаги** — последовательность действий
5. **Ожидаемый результат** — что должно произойти

## Требования к описанию тест-кейса для API-тестирования

**VCS-10905**

### Название:
```
[/v3/<new_endpoint>] [Негативный (опционально)] [<название_метода> - GET/POST/PUT/PATCH/DELETE][<Код_ответа>] <Цель_данного_api_метода>
```

**Пример:**
```
[/v3/repos/{tenant}/{project}/{repo}] [Позитивный] [Получение одного репо проекта - GET][200] - Успешное получение репо проекта
```

### Обязательные поля:
- **Название** — да (формат указан выше)
- **Описание** — да (цель, заголовки, тело, параметры)
- **Предусловие** — да (авторизация, подготовка данных)
- **Статус** — да (актуальный/на доработке/не актуален)
- **Приоритет** — нет (Blocker/Critical/Major/Minor)
- **Вид теста** — да (другое по умолчанию)
- **Вид тестирования** — да (новый функционал/регресс)
- **Автоматизирован** — да (да/нет/не применимо)
- **Папка** — да (место хранения в тестовой модели)
- **ПМИ** — да (нет по умолчанию)
- **Описание шага** — да (конкретность, краткость, единый стиль)
- **Ожидаемый результат** — да (конкретный и проверяемый)
- **CRPV/STS/SUPPORT** — да (связи с задачами)

### Формат шагов:
- Конкретность: каждый шаг проверяет конкретную функциональность
- Краткость: без упрощений и избытка деталей
- Доступность: шаги должны быть понятны другому тестировщику
- Единое оформление: безличные глаголы (нажать, ввести, перейти)

### Формат ожидаемого результата:
- Обязательность: после каждого шага
- Конкретность: конкретный и четко сформулированный
- Практичность: для API добавление curl или JSON

## Какие атрибуты тест-кейса обязательный к заполнению

### Обязательные для всех типов:

### Специфичные для API:

## Ролевая модель

**VCS-9109**

### Базовые группы привилегий:

| Код | Название |
|-----|----------|
| own | Управление пользователями |
| edit | Изменение настроек |
| read | Чтение |
| create | Создание репозиториев |
| delete | Удаление репозиториев |
| write | Запись в репозитории |
| merge_without_check | Слияние без ограничений |
| manage_comments | Управление комментариями |
| create_pr | Создание PR |
| approve_pr | Одобрение PR |
| merge_pr | Слияние PR |

### Порядок определения доступа к репозиторию:
1. Права непосредственно на репозиторий
2. При отсутствии прав на репозиторий:
   - Проект публичный? → reader
   - Группа привилегий к любому репозиторию в проекте? → права не выдаются, доступ на чтение через UI
   - Есть публичный репозиторий? → доступ
3. Если везде нет — доступа нет

### Важно:
- Привилегия write включает create_pr, approve_pr, merge_pr
- Пользователь, создавший репозиторий, получает группу owner
- Публичные репозитории доступны всем участникам тенанта

## Как создать тестовый цикл

**VCS-14134**

### Подготовка:
- Тестовый набор создан или существует ранее
- Цикл должен состоять из:
  - Тестовых кейсов нового функционала или локализации дефекта
  - Регрессионного функционала без которого тестирование нецелесообразно

### Пошаговая инструкция:
1. WORKS → Циклы → «+ Создать тестовый цикл»
2. Название: «VCS-XXXX - Название задачи»
3. Наименование релиза: версия (например: 9.6.0)
4. Тест-кейсы → Добавить подготовленные кейсы
5. Заполнить оставшиеся обязательные поля
6. Создать
7. Связать с задачей (см. VCS-14321)

## Связь тестового цикла с задачей

**VCS-14321**

### Для чего:
- Фиксация результата работы над задачей
- Быстрый поиск информации
- Проверка состава тестовых сценариев
- Подготовка к регрессу и релизам
- Оценка трудоемкости задачи
- Анализ дефектов

### Без тестового цикла задача не может считаться выполненной!

### Как связать:
1. Перейти к задаче в Task Tracker → выпадающий список «Связи»
2. Открыть тестовый цикл в новой вкладке, скопировать номер из URL
3. Нажать «Связать с задачей»
4. Вставить номер (VCS-XXXX)
5. Выбрать тип «Тестовый цикл»
6. Выбрать тип связи «Прогон к задаче»
7. Создать связь
