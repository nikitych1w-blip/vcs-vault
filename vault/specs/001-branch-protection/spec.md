# Spec: Защита веток репозитория

**Узел**: `UI / Проект / Репозитории / Репозиторий / Настройки / Защита веток` (lv6)
**Теги**: `#UI` `#api` `#v2` `#web2` `#ПАО` `#branch` `#SC`
**Версия API**: v2, web2
**Компонент**: UI + API

---

## Контекст

Администратор репозитория должен иметь возможность защищать ветки от нежелательных изменений: прямых пушей, форс-пушей, удаления. Дополнительно — требовать одобрения PR и прохождения статус-чеков перед слиянием.

---

## User Stories

### US-001: Просмотр правил защиты (P1 · #ПАО)

**Как** администратор репозитория
**Хочу** видеть список активных правил защиты веток
**Чтобы** понимать, какие ветки защищены и от чего

#### Acceptance Criteria

**AC-001.1** — Список правил
```
Given: пользователь — администратор репозитория
  And: в репозитории настроены правила защиты
 When: пользователь открывает Настройки → Защита веток
 Then: отображается список всех правил с полями: паттерн ветки, активные ограничения
```

**AC-001.2** — Пустое состояние
```
Given: пользователь — администратор репозитория
  And: правила защиты не настроены
 When: пользователь открывает Настройки → Защита веток
 Then: отображается заглушка «Правила не настроены» с кнопкой «Добавить правило»
```

**AC-001.3** — Нет доступа
```
Given: пользователь не является администратором репозитория
 When: пользователь открывает Настройки → Защита веток
 Then: секция Настройки недоступна в навигации
  And: прямой переход по URL возвращает 403
```

### US-002: Создание правила защиты (P1 · #ПАО)

**Как** администратор репозитория
**Хочу** создать правило защиты для ветки или паттерна веток
**Чтобы** предотвратить случайные прямые пуши и удаление важных веток

#### Acceptance Criteria

**AC-002.1** — Успешное создание (UI)
```
Given: пользователь — администратор репозитория
 When: пользователь открывает форму «Добавить правило»
  And: вводит паттерн «main»
  And: включает «Запретить прямой пуш»
  And: нажимает «Сохранить»
 Then: правило создано и отображается в списке
  And: пуш напрямую в main блокируется
```

**AC-002.2** — Паттерн glob
```
Given: администратор создаёт правило
 When: паттерн содержит glob «release/*»
 Then: правило применяется ко всем веткам, начинающимся с «release/»
```

**AC-002.3** — Валидация пустого паттерна
```
Given: администратор заполняет форму
 When: поле «Паттерн ветки» пустое
  And: нажимает «Сохранить»
 Then: форма не отправляется
  And: отображается ошибка «Паттерн обязателен»
```

**AC-002.4** — Конфликт паттернов
```
Given: правило с паттерном «main» уже существует
 When: администратор создаёт новое правило с паттерном «main»
 Then: API возвращает 409 Conflict
  And: UI отображает ошибку «Правило для этого паттерна уже существует»
```

### US-003: Редактирование правила (P1)

**Как** администратор репозитория
**Хочу** изменить параметры существующего правила
**Чтобы** адаптировать ограничения при изменении процесса разработки

#### Acceptance Criteria

**AC-003.1** — Успешное редактирование
```
Given: правило защиты для «main» существует с «Запретить прямой пуш»
 When: администратор открывает редактирование правила
  And: добавляет «Требовать одобрение: 2 reviewer»
  And: сохраняет
 Then: правило обновлено
  And: для слияния MR теперь требуется 2 одобрения
```

**AC-003.2** — Паттерн недоступен для изменения
```
Given: правило с паттерном «main» существует
 When: администратор открывает редактирование
 Then: поле паттерна заблокировано для редактирования
  And: отображается подсказка «Для изменения паттерна пересоздайте правило»
```

### US-004: Удаление правила (P1)

**Как** администратор репозитория
**Хочу** удалить правило защиты
**Чтобы** снять ограничения с ветки

#### Acceptance Criteria

**AC-004.1** — Удаление с подтверждением
```
Given: правило защиты для «main» существует
 When: администратор нажимает «Удалить»
 Then: отображается диалог подтверждения «Удалить правило для «main»?»
  And: после подтверждения правило удаляется
  And: прямой пуш в main снова разрешён
```

**AC-004.2** — Отмена удаления
```
Given: диалог подтверждения удаления открыт
 When: пользователь нажимает «Отмена»
 Then: диалог закрывается
  And: правило остаётся активным
```

### US-005: Блокировка прямого пуша (P1 · #ПАО)

**Как** разработчик
**Хочу** получить понятную ошибку при попытке прямого пуша в защищённую ветку
**Чтобы** понять, что нужно использовать pull request

#### Acceptance Criteria

**AC-005.1** — Блокировка через Git CLI
```
Given: ветка «main» защищена от прямых пушей
 When: разработчик выполняет «git push origin main»
 Then: push отклоняется с кодом ошибки
  And: сообщение содержит «Protected branch», название ветки, ссылку на политику
```

**AC-005.2** — Блокировка через API
```
Given: ветка «main» защищена от прямых пушей
 When: POST /api/v2/repos/{owner}/{repo}/git/refs с веткой main
 Then: ответ 403 Forbidden
  And: body: { "message": "Branch is protected" }
```

**AC-005.3** — Форс-пуш заблокирован
```
Given: ветка защищена (force_push: false)
 When: разработчик выполняет «git push --force origin main»
 Then: push отклоняется с явной ошибкой про запрет force push
```

### US-006: Требование одобрений PR (P2)

**Как** администратор репозитория
**Хочу** требовать минимальное количество одобрений перед слиянием MR
**Чтобы** обеспечить обязательное код-ревью

#### Acceptance Criteria

**AC-006.1** — MR заблокирован без одобрений
```
Given: правило требует 2 одобрения для «main»
  And: MR имеет 1 одобрение
 When: разработчик пытается слить MR
 Then: кнопка «Слить» заблокирована
  And: отображается «Необходимо ещё 1 одобрение»
```

**AC-006.2** — MR разблокируется при наборе одобрений
```
Given: правило требует 2 одобрения
 When: второй reviewer одобряет MR
 Then: кнопка «Слить» становится активной
```

### US-007: Требование статус-чеков (P2)

**Как** администратор репозитория
**Хочу** требовать прохождения CI-чеков перед слиянием
**Чтобы** предотвратить мёрж сломанного кода

#### Acceptance Criteria

**AC-007.1** — Слияние заблокировано при failing checks
```
Given: правило требует passing статус-чек «CI/build»
  And: чек «CI/build» в статусе failure
 When: разработчик пытается слить MR
 Then: слияние заблокировано
  And: UI показывает список незавершённых/упавших чеков
```

**AC-007.2** — Слияние разрешено при passing checks
```
Given: правило требует passing статус-чек «CI/build»
  And: чек «CI/build» в статусе success
 When: одобрения набраны и other requirements выполнены
 Then: кнопка «Слить» активна
```
