# FE · Компоненты (FSD + UI-кит)

Реализация страницы/фичи нового FE по `fe/design.md` и FSD ([[architecture]]).

## Куда класть код (FSD)

- **Страница** → `pages/<PageName>/` (композиция виджетов/фич).
- **Функциональность** (форма, модалка, действие) → `features/<Feature>/`.
- **Бизнес-сущность** (карточка PR, репозитория) → `entities/<Entity>/`.
- **Переиспользуемый UI** (кнопка, инпут) → `shared/ui/`.
- Внутри среза: `ui/` (компоненты), `model/` (состояние/хуки), `api/` (если нужен срез-специфичный доступ). Наружу — через `index.ts`.

## UI-кит (что использовать, не изобретать)

- **`@sds-eng/*`**: `base` (базовые компоненты), `data-grid` (таблицы/списки), `rich-textarea`, `fonts`.
- **`@primer/octicons`** — иконки.
- **`clsx`** — композиция CSS-классов.
- Готовые платформенные MFE (напр. `@sbt-works/side-menu-mfe`) — не дублировать своим кодом.

## Конвенции

- Функциональные компоненты + хуки; данные — через React Query ([[data-state]]), не прямой fetch.
- Стили — CSS рядом со срезом, проверяются Stylelint; классы — через `clsx`.
- i18n — через имеющийся механизм локализации (не хардкодить строки).
- Доступность/состояния — покрыть loading/empty/error/no-access ([[data-state]]).

## Чего избегать

- Дублировать компоненты `@sds-eng`/octicons своей реализацией.
- Класть фичу не в свой FSD-слой (ловит `steiger`).
- Хардкодить пользовательские строки вместо i18n.
