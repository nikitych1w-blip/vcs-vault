# Базовый скилл: Ручное функциональное тестирование

> Базовый контекст: SourceControl test-model и ролевая модель (SA, QA, QAA)

---

## Зоны ответственности

1. **Анализ покрытия** — инвентаризация узлов test-model, выявление `missing`/`partial`, приоритизация
2. **Написание тест-кейсов** — каждый TC привязан к узлу через `node` и `lv` в frontmatter
3. **Верификация требований** — сверка узлов модели с SA-артефактами; выявление расхождений до написания TC
4. **Ручное тестирование** — выполнение тест-кейсов по шагам с фиксацией результата
5. **Дефект-репорты** — каждый баг содержит координату узла (путь + lv + теги)
6. **Метрики покрытия** — расчёт `covered / total` по срезам: интерфейс, версия API, сущность, релиз
7. **Поддержка модели** — добавление новых узлов при выходе релизов, деактивация deprecated-узлов

---

## Жизненный цикл ручного тестирования

```
Фаза 1  Онбординг       →  Изучить FAQ, словарь, открыть vault в Obsidian
Фаза 2  Структура        →  Изучить INDEX.md, тег-систему, листовые узлы
Фаза 3  Анализ покрытия  →  Инвентаризация, gap-анализ, статусы, приоритизация
Фаза 4  Подготовка TC    →  Написание/валидация TC по формату
Фаза 5  Ручное тестирование →  Выполнение шагов, фиксация результата
Фаза 6  Отчётность       →  Дефекты, обновление статусов, метрики
Фаза 7  Контроль         →  Запрос missing/partial, ревью TC, расчёт метрик
```

### Gate перед релизом

1. Запросить у SA список изменённых узлов
2. Обновить `status` для всех затронутых узлов
3. Рассчитать метрику покрытия в срезе «изменения релиза»
4. Подтвердить: все `#ПАО`-узлы имеют `status: covered` → релиз разрешён

## Система координат (test-model)

Каждый артефакт (TC, баг, отчёт) обязан иметь координату в test-model.

### Уровни иерархии

| Уровень | Тег    | Что описывает           | Пример                                      |
|---------|--------|-------------------------|---------------------------------------------|
| lv1     | `#lv1` | Интерфейс / подсистема  | UI, API, CLI, Общие механизмы               |
| lv2     | `#lv2` | Крупный раздел          | Проект, Dashboard, v1/v2/web2               |
| lv3     | `#lv3` | Секция                  | Репозитории, Настройки                      |
| lv4     | `#lv4` | Сущность                | Репозиторий (отдельный)                     |
| lv5     | `#lv5` | Область функциональности| Код, Запросы на слияние, Создание           |
| lv6     | `#lv6` | Фича                    | Ветки, Коммиты, Теги                        |
| lv7     | `#lv7` | Конкретная операция     | Создание ветки, Открытые MR                 |
| **lv8** | `#lv8` | **Атомарный сценарий**  | **Список, Фильтры, Ограничение длины**     |

**Как читать путь:** `Интерфейс / Раздел / ... / Операция / Сценарий`

Пример: `UI / Проект / Репозитории / Репозиторий / Код / Ветки / Название / По символам` → lv8

### Теги интерфейса

| Тег  | Значение                    |
|------|-----------------------------|
| `#UI`| Веб-интерфейс               |
| `#api`| REST API                   |
| `#CLI`| Командная строка            |
| `#FE`| Frontend-специфика          |

### Версии API

| Тег   | Значение                    |
|-------|-----------------------------|
| `#v1` | REST API v1 (ПАО)           |
| `#v2` | REST API v2 (ПАО)           |
| `#v3` | REST API v3                 |
| `#web1`| Frontend API web1          |
| `#web2`| Frontend API web2 (ПАО)    |
| `#git`| Git-интеграция (ПАО)        |
| `#kafka`| Kafka-интеграция (ПАО)    |

### Теги сущностей

`#project` `#repo` `#branch` `#commit` `#tags` `#merge` `#request` `#fork` `#marker` `#pipeline` `#sonar` `#license` `#auth` `#ldap` `#jwt` `#access-token`

### Специальные теги

| Тег    | Значение                                |
|--------|-----------------------------------------|
| `#SC`  | Принадлежность продукту SourceControl   |
| `#ПАО` | Приоритетное покрытие / проверка обязательна |

### Статусы покрытия узла

| Статус      | Значение                          |
|-------------|-----------------------------------|
| `covered`   | Полное покрытиние тест-кейсами    |
| `partial`   | Покрыт частично                   |
| `missing`   | Тест-кейсов нет                   |

**Важно:** Узел с `#ПАО` и `status: missing` → **блокирует релиз**

## Формат тест-кейса (TC)

```markdown
---
id: TC-XXXX
node: UI / Проект / Репозитории / Репозиторий / Код / Ветки / Название / По символам
lv: 8
tags: [UI, SC, merge, request, FE]
api_version: web2
status: covered
priority: high
tms: TC-XXXX
---

## TC-XXXX: <название>

**Узел**: `UI / ... / Открыто / Фильтры` (lv8)
**Теги**: `#UI` `#SC` `#merge` `#request`

### Предусловия
- Пользователь авторизован
- В репозитории есть хотя бы один открытый MR

### Шаги
1. Перейти в репозиторий → «Запросы на слияние»
2. Открыть вкладку «Открыто»
3. Применить фильтр по автору
4. ...

### Ожидаемый результат
- Список MR отфильтрован по выбранному автору
- Счётчик обновился

### Альтернативные сценарии
- Нет MR по фильтру → отображается заглушка
- Сброс фильтра → список возвращается в исходное состояние

### Фактический результат
- [ ] Шаг 1: ...
- [ ] Шаг 2: ...
- [ ] Шаг 3: ...

### Комментарий тестировщика
- ...
```

## Принципы работы

- **Координата обязательна.** Любой артефакт (TC, баг, отчёт) начинается с узла: путь + lv + теги. Без координаты — невалидно.
- **Версионность API.** Тест, затрагивающий API, указывает версию. Разное поведение в разных версиях — разные TC.
- **#ПАО — приоритет.** Узлы с `#ПАО` покрываются первыми. `status: missing` на `#ПАО`-узле блокирует релиз.
- **Deprecated — вне работы.** Узлы с пометкой `(удалить раздел)` → вывести из тест-плана, деактивировать TC.
- **Сначала верификация требований.** Расхождение поведения системы с узлом модели — эскалация к SA до написания TC.
- **lv7–lv8 — основной уровень TC.** lv5–lv6 — smoke и чек-листы.
- **Ручное тестирование — по шагам.** Каждый шаг фиксируется, результат сравнивается с ожидаемым.
- **Контекст обязателен.** При описании бага указывать версию API, интерфейс, окружение.

---

## Чего избегать

- ❌ Не писать TC без координаты узла
- ❌ Не смешивать проверки разных версий API в одном TC
- ❌ Не детализировать тест только до lv1–lv2 — доводить до lv7–lv8
- ❌ Не игнорировать `#ПАО`-узлы при планировании спринта
- ❌ Не закрывать дефект без обновления `status` затронутого узла
- ❌ Не описывать баг без указания версии API/интерфейса
- ❌ Не тестировать deprecated-узлы без явного согласования с SA

- [ ] `tags` описывают интерфейс, версию API, сущности
- [ ] `priority` установлен

- [ ] Шаги понятны и воспроизводимы
- [ ] Ожидаемый результат конкретен
- [ ] Альтернативные сценарии учтены
