## Базовый скилл: `BASE-TESTING.md`

**Кому читать:** Всем QA

**Что даёт:**
- Система координат (test-model): уровни lv1–lv8, теги, статусы
- Жизненный цикл ручного тестирования
- Форматы артефактов: TC, баг-репорт, отчёт о покрытии
- Принципы работы и чеклисты
- Чего избегать при тестировании

**Когда читать:** При начале работы с ручным тестированием

**Ключевые моменты:**
- Координата узла обязательна для всех артефактов
- `#ПАО`-узлы — приоритетные, блокируют релиз при `status: missing`
- lv7–lv8 — основной уровень для функциональных TC
- Связь: узел → TC → дефект

## Скилл API: `API-SWAGGER-TESTING.md`

**Кому читать:** QA для API-тестирования

**Что даёт:**
- Анализ OpenAPI/Swagger спецификации
- Сверка контракта с SA-требованиями (spec.md)
- Формат TC для API endpoint'ов
- Чеклист ручного тестирования API
- Дефект-репорт для API-багов
- Правильные curl-шаблоны для запросов

**Когда читать:** При тестировании REST API (v1, v2, v3, web1, web2)

**Ключевые моменты:**
- Каждый TC привязан к конкретному эндпоинту и HTTP-методу
- Обязательны: 2xx + все 4xx + 5xx коды
- Аутентификация в заголовках (access-token, JWT)
- Связь: AC из spec.md → TC → OpenAPI-спецификация

**Пример TC:**
```markdown
TC-XXXX: POST /api/v2/{owner}/{repo}/resource — создание ресурса
- Сценарий 1: 201 + тело
- Сценарий 2: 400 при невалидном теле
- Сценарий 3: 401 без авторизации
- Сценарий 4: 403 без прав
```

## Скилл UI/FE: `UI-FRONTEND-TESTING.md`

**Кому читать:** QA для фронтального тестирования

**Что даёт:**
- Анализ UI-узлов test-model (lv6–lv8)
- Верификация UX-требований
- Формат TC для UI-элементов
- Чеклист ручного тестирования интерфейсов
- Дефект-репорт для UI-багов
- Паттерны проверки UI-элементов

**Когда читать:** При тестировании веб-интерфейса (web1, web2)

**Ключевые моменты:**
- Версия UI обязательна (web1 или web2)
- Каждый TC привязан к узлу test-model (path + lv)
- Обязательны: пустые состояния, ошибки, edge cases
- Скриншоты обязательны для баг-репортов
- Связь: AC из spec.md → TC → React-компоненты

**Пример TC:**
```markdown
TC-XXXX: Фильтрация MR по автору (lv8)
- Сценарий 1: Фильтрация, URL обновляется
- Сценарий A: Нет MR по фильтру → заглушка
- Сценарий B: Сброс фильтра → список возвращается
```

## Жизненный цикл тестирования (сводка)

```
1. Онбординг          → Изучить README.md → BASE-TESTING.md
2. Структура          → Пройтись по test-model/INDEX.md
3. Анализ покрытия    → Найти missing/partial узлы
4. Подготовка         → Прочитать соответствующий скилл
5. Написание TC       → Шаблоны из BASE + API/UI скилла
6. Ручное тестирование → По шагам, фиксация результата
7. Отчётность         → Баги, обновление статусов, метрики
```

## Частые вопросы

**Q: Как понять, какой скилл выбрать для тестирования?**  
A: Если тестируется REST API или CLI — `API-SWAGGER-TESTING.md`. Если веб-интерфейс — `UI-FRONTEND-TESTING.md`.

**Q: Можно ли тестировать API и UI в одном TC?**  
A: Нет — разные скиллы для разных интерфейсов. SA должен разделить AC на API и UI части.

**Q: Обязательно ли привязывать TC к узлу test-model?**  
A: Да — без координаты (node + lv) TC считается невалидным.

**Q: Как тестировать deprecated-элементы?**  
A: Только с явным согласованием с SA. Обычно deprecated-узлы исключаются из тест-плана.

**Q: Что делать, если TC не проходит, но спецификация/SA-требования тоже несовершенны?**  
A: Открыть задачу SA на обновление требований. TC не должен быть основой для требований.

## Обновление скиллов

Скиллы обновляются вручную при изменениях:
- test-model (новые уровни, теги, узлы);
- API (новые версии, эндпоинты) — сверять с `sources/vcs-api/`;
- UI (новые компоненты, паттерны).

---

## Связь с конвейером vcs-pldc

QA-скиллы применяются на стадиях схемы `vcs`:
- **qa-plan** — тест-план по `sa/specs` (узлы, сценарии, #ПАО-гейт);
- **apply** (QA-задачи) — написание TC по `BASE-TESTING`/`API-SWAGGER-TESTING`/`UI-FRONTEND-TESTING` + `patterns.md`, обновление статусов узлов.

Координата узла (`node` + `lv`) обязательна в каждом TC/баге — см. `../common/model.md`.
