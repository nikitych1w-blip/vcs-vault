# Роль: Тестировщик (QA)

> Общий контекст продукта: `common/product.md`
> Система координат: `common/model.md`

---

## Зоны ответственности

1. **Анализ покрытия** — инвентаризация узлов test-model, выявление `missing`/`partial`, приоритизация
2. **Написание тест-кейсов** — каждый TC привязан к узлу через `node` и `lv` в frontmatter
3. **Верификация требований** — сверка узлов модели с SA-артефактами (`sa/role.md`); выявление расхождений до написания TC
4. **Дефект-репорты** — каждый баг содержит координату узла (путь + lv + теги)
5. **Метрики покрытия** — расчёт `covered / total` по срезам: интерфейс, версия API, сущность, релиз
6. **Поддержка модели** — добавление новых узлов при выходе релизов, деактивация deprecated-узлов

---

## Жизненный цикл покрытия

```
Фаза 1  Онбординг       →  Изучить FAQ, словарь, открыть vault в Obsidian
Фаза 2  Структура        →  Изучить INDEX.md, тег-систему, листовые узлы
Фаза 3  Анализ покрытия  →  Инвентаризация, gap-анализ, статусы, приоритизация
Фаза 4  Написание TC     →  TC → TMS, обновить статус узла
Фаза 5  Контроль         →  Запрос missing/partial, ревью TC, расчёт метрик
Фаза 6  Поддержка        →  Отслеживать изменения, добавлять/убирать узлы
```

### Gate перед релизом

1. Запросить у SA список изменённых узлов
2. Обновить `status` для всех затронутых узлов
3. Рассчитать метрику покрытия в срезе «изменения релиза»
4. Подтвердить: все `#ПАО`-узлы имеют `status: covered` → релиз разрешён

## Формат тест-кейса

```markdown
---
id: TC-XXXX
node: UI / Проект / Репозитории / Репозиторий / Запросы на слияние / Этапы / Открыто / Фильтры
lv: 8
tags: [UI, SC, merge, request, FE]
api_version: web2
status: covered
priority: high
tms: TC-XXXX
---

## TC-XXXX: <название>

**Узел**: `UI / ... / Открыто / Фильтры` (lv8)
**Теги**: `#UI` `#SC` `#merge` `#request`

### Предусловия
- Пользователь авторизован
- В репозитории есть хотя бы один открытый MR

### Шаги
1. Перейти в репозиторий → «Запросы на слияние»
2. Открыть вкладку «Открыто»
3. Применить фильтр по автору
4. ...

### Ожидаемый результат
- Список MR отфильтрован по выбранному автору
- Счётчик обновился

### Альтернативные сценарии
- Нет MR по фильтру → отображается заглушка
- Сброс фильтра → список возвращается в исходное состояние

### Связанные узлы
- lv7: `UI / ... / Открыто`
- lv6: `UI / ... / Этапы`
```

## Принципы работы

- **Координата обязательна.** Любой артефакт (TC, баг, отчёт) начинается с узла: путь + lv + теги. Без координаты — невалидно.
- **Версионность API.** Тест, затрагивающий API, указывает версию. Разное поведение в разных версиях — разные TC.
- **#ПАО — приоритет.** Узлы с `#ПАО` покрываются первыми. `status: missing` на `#ПАО`-узле блокирует релиз.
- **Deprecated — вне работы.** Узлы `(удалить раздел)` → вывести из тест-плана, деактивировать TC.
- **Сначала верификация требований.** Расхождение поведения системы с узлом модели — эскалация к SA до написания TC.
- **lv7–lv8 — основной уровень TC.** lv5–lv6 — smoke и чек-листы.

---

## Чего избегать

- Не писать TC без координаты узла
- Не смешивать проверки разных версий API в одном TC
- Не детализировать тест только до lv1–lv2 — доводить до lv7–lv8
- Не игнорировать `#ПАО`-узлы при планировании спринта
- Не закрывать дефект без обновления `status` затронутого узла
