## Основные паттерны шагов

### Паттерн 1: Авторизация (6% TC)

**Форма:**
```
Шаг 1: [Авторизоваться / Пользователь выполнил авторизацию]
```

**Примеры:**
```
Шаг 1: Авторизоваться под <strong>Владельцем проекта</strong>
=> Успешная авторизация на платформе

Шаг 1: Пользователь выполнил авторизацию.
=> Отображается главная страница пользователя.
```

---

## Рекомендации по написанию тест-кейсов

### Что включать в каждый тест-кейс:
1. **Конкретность** - не "кликнуть кнопку", а "нажать кнопку [Название] в правом верхнем углу"
2. **Воспроизводимость** - шаги должны быть понятны другому тестировщику
3. **Конкретный результат** - не "работает", а "отображается 5 элементов"
4. **Авторизация** - указывать роль пользователя (Владелец, Ридер, Райтер)
5. **Скриншоты** - при дефектах обязательны

### Типичные паттерны имён тест-кейсов:
- **Проверка [функциональности]**
- **[Действие] [объекта]**
- **Валидация [поля/формы]**
- **Отображение [элемента]**
- **Доступность [функционала] для [роли]**

### Примеры имён:
- Проверка работоспособности пункта меню "Этапы"
- Назначение привилегии "Запись" позволяет совершать действия
- Валидация полей Название и Описание на странице создания команды
- Раздел "Команды" доступен только Владельцу проекта
- Проверка скрытия пункта "Последний вход"
- Отображение имени пользователя в пункте меню "Профиль"

## Чеклист перед выполнением тест-кейса

- [ ] Авторизация выполнена под правильной ролью
- [ ] Предусловия выполнены (существуют нужные объекты)
- [ ] Тестовые данные готовы
- [ ] Ожидаемые результаты понятны
- [ ] Известны граничные условия

## Чеклист после выполнения тест-кейса

- [ ] Фактические результаты зафиксированы
- [ ] При успехе: обновлён статус в TMS
- [ ] При дефекте: создан BUG с описанием
- [ ] Скриншоты приложены (если есть баг)
- [ ] Изменения откатились (если нужно)
