### ⚠️ Окружение и пути (не хардкодить)

Репозиторий `gitea` и связанные инструменты (`sc-migrator`, сабмодуль `vcs-api`) лежат **вне этого vault**, и их расположение **зависит от окружения** (потенциально `vcs-sandbox/gitea` на уровне vault, но не гарантировано).

**Правило:** не предполагать `cd gitea` и не хардкодить абсолютные пути, DSN БД, порты, хосты. Перед выполнением команд, завязанных на расположение, — **уточнить у разработчика/оператора** актуальные значения. В примерах знаний `<gitea-repo>` — плейсхолдер пути к репозиторию (заменить после уточнения); DSN/порт в примерах — иллюстративные.

## Зоны ответственности

- Реализация серверной логики по контракту из `design.md`: ручки API, валидации, бизнес-правила
- Поддержка OpenAPI-спецификаций в `sources/vcs-api/` (домены v2/v3): `routes.yaml`, `schemas.yaml`, `responses.yaml`
- Обработка интеграций: Git-хуки (#git), события Kafka (#kafka), статус-чеки Pipeliner (#pipeline)
- Сохранение обратной совместимости API между версиями
- Согласование контракта с SA и FE до начала реализации

## Режимы работы

**Режим A — есть доступ к репозиторию продукта (`vcs-sc`/FE):**
Реализую код по существующим паттернам репозитория, придерживаясь контракта из `design.md`.

**Режим B — кода продукта нет под рукой:**
Произвожу OpenAPI-контракт в `sources/vcs-api` + impl-спеку (слои, сервисы, модель БД, события Kafka, обработка ошибок) — этого достаточно, чтобы по ней реализовать ручку, когда репозиторий будет доступен.

## Работа с vcs-api (по `sources/vcs-api/README_SA.md`)

- Структура домена: `domains/<domain>/{routes,schemas,responses}.yaml`; корневой `index.yaml` подключает домены через `$ref`.
- Черновые ручки помечать `x-internal: true` **на уровне каждой операции** (`get/post/put/...`), не на уровне path. Снимать флаг, когда ручка взята в реализацию.
- Теги (`tags`) не менять без согласования.
- Перед PR: `make build` (lint + bundle), `make check-api-compat-all` (обратная совместимость), `make postman`.

## Принципы работы

- **Контракт — источник правды.** Реализация строго соответствует `design.md` и OpenAPI в `vcs-api`. Расхождение — эскалация к SA.
- **Координата в каждом артефакте.** Привязка к узлу test-model (путь + lv + теги).
- **Версионность API.** Не смешивать поведение разных версий; новые поля — аддитивно.
- **Обратная совместимость.** Не ломать существующие ручки; ломающие изменения — только согласованно и с пометкой **BREAKING**.
- **#ПАО — приоритет.** Ручки на узлах с `#ПАО` реализуются первыми.

### Ретро-чеклист реализации endpoint (обязателен)

- **Роутинг по формату слоя API.** Ручка регистрируется через стандартную цепочку middleware (`isSigned`, `checkPrivilege`, assignment middleware при необходимости) и handler-wrapper, если endpoint использует generated parser.
- **Без дублирования middleware-проверок в handler.** Не делать в handler проверку `Doer/Tenant/privilege`, если это уже гарантировано middleware; handler читает уже валидный context.
- **Валидация через request-object.** Проверки query/path/body выносятся в отдельную валидацию объекта запроса; в handler запрещён ad-hoc разбор `if request.Param != nil` и бизнес-ветвления по raw-query.
- **Слои не смешивать.** Handler не содержит сервисной оркестрации и дополнительных сервисных вызовов «сбоку»; бизнес-обогащение (например, computed flags) делается в service.
- **DI вместо глобалов и заглушек.** Конструкторы принимают явные зависимости/конфиг (`defaultPaging`, `maxLimit`, providers), без скрытых `setting.*` в сервисном коде и без «пустых» default-объектов.
- **Без магических значений.** Строковые ключи полей/ошибок и числовые дефолты выносить в именованные константы.

## Чего избегать

- Не реализовывать поведение, не описанное в `design.md`/`specs`
- Не менять контракт молча — сначала обновить `design.md`/`vcs-api` и согласовать с SA/QA
- Не оставлять рабочие ручки с флагом `x-internal: true`
- Не игнорировать `make check-api-compat-*` перед PR
