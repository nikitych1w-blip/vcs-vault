## Зоны ответственности

1. **Анализ UI-узлов** — изучение структуры экранов, фич, операций в test-model
2. **Верификация UX-требований** — сверка поведения интерфейса с AC в spec.md
3. **Написание TC для UI** — каждый TC привязан к узлу UI через `node` и `lv`
4. **Ручное тестирование интерфейсов** — клики, навигация, заполнение форм
5. **Проверка адаптивности** — desktop, tablet, mobile (если применимо)
6. **Тестирование состояний** — пустые состояния, загрузка, ошибки
7. **Дефекты UI** — баги с указанием экрана, элемента, версии API (web1/web2)
8. **Импорт из TMS** — парсинг тест-кейсов из Excel-экспорта TMS

## Жизненный цикл UI тестирования

```
Фаза 1  Онбординг         →  Изучить vault/test-model/1. UI/, структуру экранов
Фаза 2  Анализ покрытия   →  Инвентаризация узлов lv6–lv8, выявление missing
Фаза 3  Согласование с SA  →  Проверка, что все AC покрываются UI-тестами
Фаза 4  Подготовка TC     →  Написание TC по формату для UI-узлов
Фаза 5  Импорт в TMS      →  Импорт TC из Markdown в систему управления тестами
Фаза 6  Ручное тестирование →  Открытие экранов, выполнение сценариев
Фаза 7  Отчётность        →  Дефекты, обновление статусов, метрики
```

## TC-XXXX: Фильтрация MR по автору

**Узел**: `UI / Запросы на слияние / Запросы / Открыто / Фильтры / По автору` (lv8)
**Версия API**: web2
**Теги**: `#UI` `#SC` `#merge` `#request`

### Предусловия
- Пользователь авторизован
- В репозитории есть хотя бы 3 MR от разных авторов (user1, user2, user3)

### Шаги
1. Открыть репозиторий
2. Перейти на вкладку «Запросы на слияние»
3. Открыть вкладку «Открыто»
4. Нажать кнопку «Фильтры»
5. В разделе «Автор» выбрать из выпадающего списка → `user2`
6. Нажать «Применить»

### Ожидаемый результат
- В списке отображается только 1 MR (от author=user2)
- Счётчик MR обновился на 1
- В URL добавлен параметр `author=user2`
- В списке фильтров отображается активный фильтр «Автор: user2»

### Альтернативные сценарии

#### Сценарий A: Нет MR по фильтру
**Шаги:** Выбрать автора, у которого нет MR
**Ожидаемый результат:**
- Отображается заглушка: «Нет запросов на слияние»
- Кнопка «Сброс» активна
- Счётчик MR = 0

#### Сценарий B: Сброс фильтра
**Шаги:** После применения фильтра нажать «Сброс»
**Ожидаемый результат:**
- Список возвращается к исходному состоянию (все MR)
- Параметр `author` удаляется из URL
- Активный фильтр исчезает из списка

- [ ] Кнопка «Фильтры» отображается

## Паттерны UI тестирования

### Элементы интерфейса

| Элемент | Что проверять |
|---------|---------------|
| Кнопка | Доступна? Активна? Подсветка при hover? |
| Поле ввода | Валидация? Placeholder? Ошибки? |
| Выпадающий список | Открывается? Выбор работает? |
| Табы | Переключение? Активный таб подсвечивается? |
| Модальное окно | Закрывается? Кнопки работают? |
| Заглушка | Появляется при пустом списке? Дизайн верный? |
| Лоадер | Исчезает после загрузки? Блокирует клики? |

### Состояния

| Состояние | Что проверять |
|-----------|---------------|
| Пустой список | Заглушка, кнопка создания, дизайн |
| Загрузка | Лоадер, блокировка, прогресс |
| Ошибка | Сообщение, кнопка повтора, логи |
| Успех | Toast, иконка, переход |
| Состояние загрузки | Прогресс-бар, остаток времени |

### Навигация

| Путь | Что проверять |
|------|---------------|
| Меню | Открывается/закрывается, пункты |
| Хлебные крошки | Рабочие ссылки, путь верный |
| URL | Обновляется, bookmarks работают |
| Back/Forward | Состояние сохраняется |

## Принципы UI тестирования

- **Координата обязательна.** Каждый TC привязан к узлу test-model (path + lv + теги)
- **Версия UI обязательна.** web1 и web2 — отдельные тесты
- **Шаги воспроизводимы.** Каждый шаг должен быть понятен другому тестировщику
- **Ожидаемый результат конкретен.** Не «работает», а «отображается 5 элементов»
- **Альтернативные сценарии обязательны.** Пустой список, ошибки, edge cases
- **Скриншоты важны.** Баг без скриншота — сложнее воспроизвести
- **UI-элементы проверять отдельно.** Не только функционал, но и UI-контролы

---

## Чего избегать

- ❌ Не писать TC без координаты узла (node + lv)
- ❌ Не тестировать только happy path — обязательны ошибки и пустые состояния
- ❌ Не смешивать web1 и web2 в одном TC
- ❌ Не описывать баг без скриншота
- ❌ Не игнорировать состояния лоадера (ожидание не завершается)
- ❌ Не тестировать deprecated-элементы без согласования
- ❌ Не писать «кликнуть на кнопку» — писать «нажать кнопку [Создать] в правом верхнем углу»

## Специфика web2 (Новый FE)

### Особенности тестирования

| Особенность | Как тестировать |
|-------------|-----------------|
| SPA-навигация | URL обновляется без перезагрузки |
| React-состояния | Лоадеры, error boundaries, suspense |
| GraphQL API | Запросы в Network tab (GraphQL), не REST |
| TypeScript | Ошибки в консоли при рендеринге |

### Проверка GraphQL запросов (web2)

```graphql
# В Chrome DevTools → Network → filter: graphql
query {
  repository(owner: "owner", name: "repo") {
    pullRequests(state: OPEN, first: 20) {
      nodes {
        id, title, author { login }
      }
    }
  }
}
```

**Что проверять:**
- Запрос приходит корректный
- Ответ содержит все поля
- Ошибки возвращаются в `errors` массиве
- Pagination работает через `pageInfo { hasNextPage, endCursor }`

## Импорт тест-кейсов из TMS (Excel)

### Структура Excel-файла

Тест-кейсы экспортируются из TMS в формате Excel. Структура строк:

| Кол-во полей | Описание |
|--------------|----------|
| 46-48 | Начало тест-кейса + шаг 1 (unit, summary, description, step 1) |
| 15-17 | Шаги 2, 3, 4... без unit (продолжение предыдущего TC) |

**Поля (индексы):**
- `[0]` unit - ID тест-кейса
- `[1]` summary - Название
- `[2]` description - Описание TC
- `[11]` priority - Приоритет
- `[12]` test_step - Номер шага
- `[13]` step_type - Тип шага (step_by_step)
- `[14]` step_desc - Описание шага
- `[16]` step_result - Ожидаемый результат
- `[24]` status - Статус
- `[35]` folder - Путь в TMS
- `[42]` test_level - Уровень теста (UI/API)

### Парсинг Excel

Используйте скрипт `parse_xlsx.py` для извлечения тест-кейсов:

```bash
python parse_xlsx.py \
    export_test_cases_*.xlsx \
    --output test_cases_export.json \
    --summary
```
