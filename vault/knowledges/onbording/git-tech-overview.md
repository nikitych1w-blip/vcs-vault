# Часть I. Git Core

Фундамент. Пока не уложится, как устроены blob/tree/commit/tag на уровне байтов, всё остальное (pack, refs, Gitaly) висит в воздухе.

## 1.1 Объектная модель

### Протокол — формат объектов побайтово

Каждый Git-объект на диске — одна и та же конструкция:

```
<type> <size>\0<content>
```

Всё это сжимается zlib (DEFLATE). Ключевой момент: **хеш считается по несжатому виду** — по строке `<type> <size>\0<content>`, а не по сжатым байтам.

```
sha1( "blob 11\0Hello World" ) = SHA объекта
```

**blob** — сырое содержимое файла. Никаких метаданных: ни имени, ни прав, ни даты.

```
blob 11\0Hello World
```

Имя и права хранятся в дереве, а не в блобе. Поэтому два одинаковых файла с разными именами = **один** blob. Это база дедупликации во всём Git.

**tree** — последовательность записей вида:

```
<mode> <name>\0<raw_sha>
```

Критичная деталь: **SHA в дереве хранится в сыром бинарном виде** — 20 байт (SHA-1) или 32 байта (SHA-256), не hex. А в коммите (ниже) — наоборот, hex. Разное представление в разных объектах, классический источник багов в самописных парсерах.

Mode — ASCII в восьмеричной записи:

```
100644  обычный файл
100755  исполняемый файл
120000  симлинк
160000  gitlink (submodule — указывает на commit чужого репо)
40000   директория (поддерево)
```

Директория — `40000`, без ведущего нуля (Git печатает через `%o`, 040000 в octal печатается как 40000). Файлы — 6 цифр, директории — 5. Ещё одна ловушка.

Записи **отсортированы по имени**, причём поддеревья сортируются так, будто к имени дописан `/`. Порядок строгий — от него зависит SHA дерева.

**commit** — текстовый формат:

```
tree <40-hex-sha>
parent <40-hex-sha>      ← ноль или больше строк
author Name <email> <unix_ts> <tz>
committer Name <email> <unix_ts> <tz>

commit message
```

- 0 parent → root commit
- 1 parent → обычный коммит
- 2+ parent → merge (octopus = 3+)
- `author` ≠ `committer`: author — кто написал, committer — кто применил. При rebase/cherry-pick author сохраняется, committer меняется. Поэтому rebase меняет SHA даже при том же содержимом.
- Для подписи добавляется заголовок `gpgsig` прямо в тело объекта → подпись меняет SHA коммита.

**tag (аннотированный)**:

```
object <sha>
type commit
tag v1.0.0
tagger Name <email> <ts> <tz>

release notes
```

Lightweight-тег — это **не** объект, а просто ref на коммит. Аннотированный — полноценный объект со своим SHA.

### SHA-1 → SHA-256

После атаки **SHAttered** (2017, реальная коллизия SHA-1) Git по умолчанию использует **SHA-1DC** (collision detection): детектит SHAttered-подобные структуры и падает с ошибкой вместо записи коллизии.

Параллельно добавлен полноценный SHA-256 (`git init --object-format=sha256`): хеши 32 байта / 64 hex. Репозиторий целиком либо SHA-1, либо SHA-256 — смешивать нельзя. Полная interop между двумя форматами — давняя цель, но на практике пока ограничена, поэтому массовой миграции в индустрии не случилось.

### Loose objects и zlib

Свежезаписанный объект кладётся «россыпью» (loose):

```
.git/objects/ab/cdef0123...    ← ab = первые 2 hex SHA, остальное — имя файла
```

256 поддиректорий по первому байту — чтобы не упереться в лимит файлов на директорию у старых ФС.

Сжатие zlib DEFLATE, два разных конфига:

- `core.loosecompression` — для loose, по умолчанию уровень **1** (скорость): объекты временные, скоро уедут в pack
- `pack.compression` — для pack, по умолчанию **6**: pack живёт долго, важнее коэффициент

Осознанный trade-off: пишем быстро и дёшево, потом перепаковываем плотно.

### Архитектура и различия платформ

**GitHub — object pools / alternates.** Тысячи форков `torvalds/linux` не хранят по копии истории. Корень fork-сети держит общий пул объектов, каждый форк через `objects/info/alternates` ссылается на него и хранит только свои уникальные объекты. GC и перепаковка идут на уровне fork-сети. Без этого данные физически не уместить.

**GitLab / Gitaly.** Gitaly **не меняет** формат на диске — те же стандартные loose/packed объекты в обычном git-репо. Gitaly добавляет сверху gRPC-слой: доступ к объектам через `RepositoryService`, `CommitService`, `BlobService`. Абстрагируется не хранение, а *доступ* — чтобы навесить кэш, репликацию (Praefect) и вынести git-операции на отдельные ноды.

**Bitbucket — Mesh.** Раньше Bitbucket DC держал репозитории на общем **NFS** — единая точка контеншена. Mesh шардит репозитории по нодам, каждая держит полную копию своих репо, запись реплицируется на кворум. NFS-бутылочное горло уходит, появляется HA.

**Gitea — напрямую.** Дёргает бинарник `git` через `os/exec` поверх локальной (или общей) ФС. Никакого слоя абстракции — просто и быстро на одной ноде. Для горизонтального масштабирования нужна общая ФС (NFS/CephFS), а это возвращает ровно те проблемы, ради которых GitLab сделал Gitaly, а Atlassian — Mesh. **Это и есть причина, по которой мы прикручиваем Gitaly к Gitea.**

### Граничные случаи на масштабе

- **Миллионы loose-объектов** до GC: 256 поддиректорий перестают спасать, обход объектов захлёбывается в inode/IO.
- **Огромные деревья** (моно-репо со 100K+ файлов в одной директории): дерево как объект становится мегабайтным, любая операция целиком его распаковывает.
- **Глубокие fork-сети**: один большой объект в общем пуле раздувает хранилище всех форков сразу.
- **Объекты-сироты после force push**: достижимы через reflog, место не освобождается до истечения reflog (90 дней) + GC.

### Trade-off, зашитые в дизайн

- **Контентная адресация** даёт бесплатную дедупликацию и целостность, но любое изменение истории = новые SHA вниз по цепочке (боль rebase общих веток).
- **Снапшоты, а не дельты**: концептуально Git хранит полные снапшоты дерева на каждый коммит. Дельты — деталь *упаковки*, не модели. Упрощает логику ценой зависимости от хорошего GC.
- **Loose=быстро / pack=плотно**: оптимизация под «много мелких записей → редкая упаковка».

### Код, который стоит открыть

- `git/object.h`, `git/cache.h` — определения типов
- `git/tree-walk.c` — парсинг дерева (raw SHA + mode)
- `git/commit.c` — `parse_commit_buffer()`
- `git/object-file.c` (ранее `sha1-file.c`) — запись/чтение loose, zlib
- Gitaly: `proto/blob.proto`, `proto/commit.proto`
- Gitea: `modules/git/` — тонкий слой над `os/exec`

## 1.2 Pack-файлы и дельта-компрессия

Loose-объекты не масштабируются: миллион файлов = миллион inode и катастрофа по IO. Pack — это плотная упаковка многих объектов в один файл с дельта-сжатием.

### Протокол — формат .pack и .idx

**Файл `.pack`:**

```
"PACK"            4 байта — сигнатура
<version>         4 байта — 2 или 3 (почти везде 2)
<object count>    4 байта — число объектов
<object>...       сами объекты подряд
<trailer>         20/32 байта — SHA всего пака (контроль целостности)
```

Каждый объект в паке начинается с **переменной длины заголовка**: первый байт — старший бит «есть продолжение», следующие 3 бита — тип, остаток — младшая часть размера; далее varint-байты размера. Затем zlib-сжатые данные.

Типы внутри пака:

```
OBJ_COMMIT     = 1
OBJ_TREE       = 2
OBJ_BLOB       = 3
OBJ_TAG        = 4
OBJ_OFS_DELTA  = 6   ← дельта со смещением на базу
OBJ_REF_DELTA  = 7   ← дельта по SHA базы
(5 зарезервирован/не используется)
```

**Дельта-объекты.** Вместо полного содержимого хранится разница от базового объекта:

- **OFS_DELTA** — база указана **отрицательным смещением** (varint) от текущей позиции в этом же паке. Компактнее, но работает только если база в том же паке.
- **REF_DELTA** — база указана 20/32-байтным **SHA**. База может быть в другом паке (нужно для thin-pack при передаче по сети).

Тело дельты: размер источника (varint), размер цели (varint), затем инструкции — `copy` (скопировать диапазон из базы: offset+size) и `insert` (вставить N сырых байт). Так Git описывает «возьми кусок старого файла, вставь сюда новый кусок».

**Поиск дельта-базы** при упаковке управляется двумя параметрами:

- `--window` (по умолчанию 10) — сколько объектов держать в окне сравнения
- `--depth` (по умолчанию 50) — максимальная длина цепочки дельт

Глубокая цепочка экономит место, но замедляет чтение: чтобы развернуть объект, нужно развернуть всю цепочку до полной базы. Это прямой trade-off «диск против CPU при чтении».

**Файл `.idx` (версия 2):**

```
\377tOc            4 байта — магия (версия 2)
<version>          4 байта — = 2
fanout[256]        256×4 байта — кумулятивные счётчики по первому байту SHA
sorted SHAs        отсортированные SHA всех объектов
CRC32[]            контрольные суммы каждого объекта
offsets[]          смещения объектов в .pack
large offsets[]    для паков > 2 ГБ
trailers           SHA пака + SHA индекса
```

**Fanout-таблица** — ключ к скорости. `fanout[b]` = сколько объектов с первым байтом ≤ b. Зная первый байт искомого SHA, сразу получаешь диапазон в отсортированном массиве и делаешь бинарный поиск только в нём. Поиск объекта = O(log n) без чтения самого пака.

### Индексы поверх паков

- **commit-graph** (`.git/objects/info/commit-graph`) — предвычисленный граф коммитов: родители, дата, **generation numbers**. Резко ускоряет `git log`, `merge-base`, подсчёт ahead/behind, потому что не нужно парсить каждый commit-объект.
- **Multi-pack-index / MIDX** (`.git/objects/pack/multi-pack-index`) — единый индекс поверх **нескольких** паков. Без него поиск объекта = перебор всех `.idx`. С MIDX — один lookup. Критично, когда паков десятки.
- **Reachability bitmaps** (`.bitmap` рядом с паком) — для ключевых коммитов хранят битовую маску всех достижимых объектов (сжатие EWAH). Превращают «обойти весь граф, чтобы понять что отправить при clone» в «прочитать готовый битмап». Именно это даёт GitHub быстрый clone на огромных репозиториях.

### Жизненный цикл и GC

```
push → loose objects → git gc → pack + idx → repack -a -d → один большой пак
```

- `git gc` — упаковка, удаление недостижимого (с учётом reflog), построение индексов.
- `git repack -a -d` — переупаковать всё в один пак и удалить старые.
- **Cruft packs** (Git 2.38+) — отдельный пак для **недостижимых** объектов с файлом `.mtimes`, хранящим время последнего обращения к каждому. Раньше недостижимые объекты «рассыпались» обратно в loose с текущим mtime — на большом репо это лавина loose-объектов. Cruft pack держит их упакованными и корректно отсчитывает grace-период до удаления.

### Различия платформ

- **GitHub** агрессивно использует bitmaps и собственный планировщик обслуживания репозиториев; перепаковка идёт на уровне fork-сети (общий пул), чтобы битмапы покрывали популярные коммиты родителя.
- **Gitaly** управляет упаковкой через housekeeping-логику и умеет кэшировать результат `pack-objects` (см. 7.1), чтобы повторные clone не пересобирали пак.
- **Bitbucket Mesh** реплицирует уже готовые паки между нодами; перепаковка локальна для ноды, но должна согласовываться с репликацией.
- **Gitea** полагается на стандартный `git gc` (есть встроенный планировщик housekeeping), запускаемый по расписанию или вручную через админку.

### Граничные случаи

- **GC во время clone**: если `repack -d` удалит пак, который читает идущий clone, спасает то, что Git держит открытый файловый дескриптор; но «pack disappeared» всё же случается при агрессивных настройках — отсюда осторожность с `gc` под нагрузкой.
- **Слишком глубокие дельты** делают чтение «холодных» объектов дорогим — на серверах depth иногда ограничивают.
- **Распухший cruft pack**: если постоянно генерируется мусор (форс-пуши, перезаписи), grace-период держит его на диске и место не освобождается.
- **Пак > 4 ГБ**: версия 2 индекса требует large-offset механизма; самописные парсеры часто это не учитывают.

### Trade-off

- **Дельта-глубина**: место против CPU/латентности чтения.
- **Битмапы**: дополнительное место и стоимость построения против радикального ускорения clone/fetch.
- **Частый GC**: чистое хранилище и быстрые операции против нагрузки на IO/CPU в момент GC и риска гонок под нагрузкой.

### Код

- `git/pack-objects.c`, `git/builtin/pack-objects.c` — упаковка и поиск дельт
- `git/packfile.c` — чтение паков, разворачивание дельт
- `git/pack-bitmap.c` — reachability bitmaps
- `git/midx.c` — multi-pack-index

## 1.3 Ref-система и её эволюция

Объекты — это контент. Refs — это человекочитаемые указатели на коммиты (ветки, теги). Именно по refs пользователь и платформа ориентируются в истории.

### Протокол — три поколения хранения refs

**1. Loose refs.** Один файл на ref:

```
.git/refs/heads/main         содержит 40 hex SHA + \n
.git/refs/tags/v1.0          содержит SHA тега или коммита
```

Просто, но при тысячах веток — тысячи файлов и дорогой `git branch --list` (надо stat-ить каждый).

**2. Packed-refs.** Все refs в одном файле `.git/packed-refs`:

```
# pack-refs with: peeled fully-peeled sorted
<sha> refs/heads/main
<sha> refs/tags/v1.0
^<peeled-sha>                ← для аннотированного тега: на какой коммит он указывает
```

`^`-строка («peeled») — закешированный коммит, на который ссылается аннотированный тег, чтобы не разворачивать tag-объект каждый раз. Loose-ref имеет приоритет над packed (свежая запись «перекрывает» упакованную).

**Обновление ref** атомарно через lock-файл:

```
создать refs/heads/main.lock  →  записать новый SHA  →  atomic rename поверх main
```

Это даёт атомарность на уровне одной ref, но порождает классическую **гонку TOCTOU** при параллельных push в одну ветку (см. ниже).

**3. Reftable** (бинарный формат, портирован из JGit/Gerrit; в Git proper доступен как экспериментальный backend в свежих версиях — *сверяйте конкретную версию*). Блочный бинарный формат с префиксным сжатием имён и встроенным reflog. Решает две боли разом: миллионы refs (моно-репо, тысячи PR-ветвей) и атомарность пакетных обновлений. Чтение конкретного ref = поиск по блокам, а не парсинг мегабайтного `packed-refs`.

### reflog

Отдельно от самих refs:

```
.git/logs/refs/heads/main
.git/logs/HEAD
```

Каждая строка — «было SHA → стало SHA, кто, когда, действие». Reflog — это то, что делает «потерянные» после force push/reset коммиты восстановимыми (`git reflog`, `git reset --hard HEAD@{2}`) и держит их недостижимые объекты живыми до истечения срока (по умолчанию 90 дней для достижимых, 30 для недостижимых).

### Архитектура и различия платформ

**GitHub — fork networks и shared refs.** Форки делят пул объектов, но refs у каждого свои. Специальные ref-неймспейсы (вроде `refs/pull/*`) хранят PR-состояние. Перепаковка refs координируется с object pool родителя.

**GitLab / Gitaly.** Refs читаются/пишутся через RPC (`RefService`: перечисление веток/тегов, `OperationService`: создание/удаление через серверный git). Критично: **обновления refs в Gitaly транзакционны** — через reference-transaction хук Praefect организует голосование реплик (см. 3.2), чтобы primary и secondaries согласились на новое значение ref до коммита.

**Bitbucket.** Branch permissions enforced на уровне обновления ref: перед применением проверяется, можно ли этому пользователю двигать данную ветку. Mesh должен согласовать обновление ref между репликами.

**Gitea — ref namespaces для изоляции.** На одной ноде Gitea держит репозитории многих пользователей; изоляция — на уровне путей и неймспейсов refs внутри каждого репозитория. PR хранятся в служебном неймспейсе (см. 5.1).

**Symbolic refs.** `HEAD`, `ORIG_HEAD`, `MERGE_HEAD`, `FETCH_HEAD` — указатели на другие refs или служебные состояния. `HEAD` обычно символический (`ref: refs/heads/main`); при detached HEAD содержит сам SHA. Платформы используют `HEAD` репозитория как «ветку по умолчанию», показываемую при clone.

### Граничные случаи

- **TOCTOU при параллельных push**: два клиента одновременно пушат в `main` от одного и того же старого SHA. Оба проходят проверку «old соответствует текущему», но применить можно только одно обновление. Решается атомарным compare-and-swap при обновлении ref (старое значение должно совпасть в момент записи) — это и есть серверная проверка fast-forward.
- **Миллионы refs**: `packed-refs` мегабайтного размера читается целиком на каждую операцию → reftable.
- **Битый lock-файл**: упавший процесс может оставить `*.lock`, блокирующий ветку, — операционная боль, требующая очистки.
- **Гонка loose vs packed**: одновременная упаковка refs и обновление одной из них.

### Trade-off

- **loose**: простота против дороговизны массовых операций.
- **packed-refs**: быстрое перечисление против неатомарности пакетных обновлений и линейного чтения.
- **reftable**: масштаб и транзакционность против сложности формата и новизны (меньше инструментов умеют читать напрямую).

### Код

- `git/refs.c`, `git/refs/files-backend.c` — loose/packed backend
- `git/refs/reftable-backend.c` — reftable backend (в свежих версиях)
- `git/builtin/pack-refs.c` — упаковка refs
- Gitaly: `proto/ref.proto`, reference-transaction логика в `internal/`
- JGit `org.eclipse.jgit.internal.storage.reftable` — эталонная реализация reftable

# Часть II. Протоколы передачи данных

Как байты из `.git` одного компьютера попадают в `.git` другого. Один и тот же набор операций (`upload-pack` для отдачи, `receive-pack` для приёма) работает поверх трёх транспортов: native git://, SSH, HTTP.

## 2.1 Git Protocol v1 и v2

### Протокол — pkt-line

Базовый кирпич всех git-протоколов — **pkt-line**. Каждая строка предваряется 4 hex-символами длины (включая сами эти 4 байта):

```
0009done\n          ← 0x0009 = 9 байт всего: "0009" + "done\n"
003f<sha> refs/...   ← 0x003f = 63 байта строки
0000                 ← flush-pkt: конец секции
0001                 ← delim-pkt (только v2): разделитель секций
0002                 ← response-end-pkt
```

- Максимум данных в одной pkt-line — 65516 байт (вся строка ≤ 65520).
- `0000` (flush) — главный разделитель фаз диалога.
- Спецзначения `0001`/`0002` появились в v2.

### Protocol v1 — ref advertisement + negotiation

Диалог clone/fetch:

1. **Ref advertisement.** Сервер шлёт список всех refs. В первой строке после первого ref через `\0` идёт список **capabilities** сервера:

```
<sha> HEAD\0multi_ack thin-pack side-band-64k ofs-delta agent=git/2.x
<sha> refs/heads/main
<sha> refs/tags/v1.0
0000
```

1. **Negotiation.** Клиент объявляет, что хочет и что уже имеет:

```
want <sha> multi_ack side-band-64k ofs-delta\n
want <sha>\n
0000
have <sha>\n          ← коммиты, которые у клиента уже есть
have <sha>\n
0000
done\n
```

1. Сервер по `have`/`want` вычисляет минимальный набор объектов и шлёт pack.

**Алгоритм negotiation** — сердце эффективного fetch. Клиент порциями шлёт `have` (двигаясь по своей истории вглубь), сервер отвечает `ACK <sha>` при общем коммите или `NAK`. Capability `multi_ack`/`multi_ack_detailed` позволяет находить **несколько** общих точек и не тащить лишнего. Цель — найти границу «общего» предка и переслать только то, чего у клиента нет.

**Side-band.** При `side-band-64k` ответ мультиплексируется по каналам:

- канал 1 — данные пака
- канал 2 — прогресс (то, что видно как «Receiving objects… 42%»)
- канал 3 — ошибки

Это позволяет одновременно лить пак и показывать прогресс в одном потоке.

### Protocol v2 — командный, stateless-friendly

Включается через переменную окружения `GIT_PROTOCOL=version=2` (по SSH/HTTP она пробрасывается). Принципиальные отличия:

- **Нет принудительного ref advertisement.** Сервер сначала объявляет только свои capabilities и список доступных команд. Клиент сам запрашивает то, что нужно.
- **Команда `ls-refs`** с фильтрами (префикс, паттерн) — можно спросить только нужные refs, а не весь список. На репозитории с сотнями тысяч refs это разница между мгновенным и многосекундным ответом.
- **Команда `fetch`** с явными аргументами (`want`, `have`, `done`, `filter`, `deepen` и т.д.).
- **Stateless по дизайну** — отлично ложится на HTTP, где каждый запрос независим.

```
# v2: клиент запрашивает только ветки
0014command=ls-refs
0001
0009peel
000bref-prefix refs/heads/
0000
```

Зачем v2: на больших репозиториях фаза advertisement в v1 пересылала **все** refs до того, как клиент скажет, что ему надо. v2 убирает этот налог.

### Архитектура и различия платформ

- **GitHub** реализует серверные оптимизации negotiation и повсеместно использует bitmaps (1.2), чтобы вычислять набор объектов без обхода графа.
- **Gitaly** обслуживает `upload-pack`/`receive-pack` как RPC (`SmartHTTPService`, `SSHService`) и применяет **sidechannel** для отдачи пака: тело пака идёт мимо обычного gRPC-сообщения (через отдельный поток), чтобы не платить накладные расходы протобуфа на гигабайтах. Это важная деталь именно для нашей интеграции.
- **Bitbucket** проксирует git-протокол через свой слой, маршрутизируя к нужной Mesh-ноде.
- **Gitea** для HTTP отдаёт управление либо `git-http-backend`, либо собственному обработчику; negotiation выполняет сам git-процесс.

### Граничные случаи

- **Clock skew при negotiation**: negotiation основан на достижимости, а не на времени, поэтому рассинхрон часов сам по себе fetch не ломает; но он портит сортировку коммитов по дате и эвристики (например, «остановиться, углубившись на N дней»), из-за чего диалог бывает менее эффективным.
- **Гигантский advertisement в v1**: репозитории с сотнями тысяч refs (PR-ветви) — главный мотиватор перехода на v2.
- **Обрыв side-band**: ошибка приходит по каналу 3 уже посреди потока — клиент должен корректно прервать приём пака.
- **Stateless HTTP + длинная negotiation**: каждый раунд — отдельный POST; на «далёких» историях это много round-trip’ов.

### Trade-off

- **v1 простота** против налога на advertisement.
- **v2 эффективность** против большей сложности (состояние диалога переехало в явные команды).
- **side-band**: мультиплекс прогресса против усложнения парсинга потока.

### Код

- `git/pkt-line.c` — формат pkt-line
- `git/upload-pack.c`, `git/builtin/receive-pack.c` — серверные стороны
- `git/fetch-pack.c`, `git/send-pack.c` — клиентские стороны
- `git/serve.c`, `git/protocol-v2` логика — диспетчер команд v2
- `git/Documentation/technical/` — `pack-protocol.txt`, `protocol-v2.txt` (первоисточник)
- Gitaly: `proto/smarthttp.proto`, `proto/ssh.proto`

## 2.2 SSH-транспорт

Самый распространённый транспорт для аутентифицированного доступа. Быстрый, потому что бинарный и без HTTP-обёрток.

### Протокол — как SSH запускает git

`git clone ssh://git@host/repo.git` под капотом:

1. Клиент поднимает SSH-сессию к `git@host`.
1. Сервер по SSH запускает команду: `git-upload-pack 'repo.git'` (для clone/fetch) или `git-receive-pack 'repo.git'` (для push).
1. stdin/stdout SSH-канала становятся транспортом для pkt-line диалога из 2.1.

Три серверные команды:

- `git-upload-pack` — отдаёт объекты (clone/fetch)
- `git-receive-pack` — принимает объекты (push)
- `git-upload-archive` — отдаёт архив (`git archive --remote`)

### Единый пользователь и ForceCommand

Ключевой трюк платформ: миллионы пользователей, но **один системный аккаунт** `git@`. Как различать, кто пушит, и не дать выполнить произвольную команду?

**Вариант A — `command=` в authorized_keys.** Каждому публичному ключу сопоставляется принудительная команда:

```
command="gitlab-shell key-123",no-port-forwarding,no-X11-forwarding,no-pty ssh-rsa AAAA...
```

Что бы клиент ни попросил — выполнится только `gitlab-shell key-123`. Идентичность пользователя зашита в аргумент (`key-123`). Внутри обёртка читает оригинальную команду из переменной `SSH_ORIGINAL_COMMAND`, валидирует (только upload/receive-pack) и проксирует.

**Вариант B — `AuthorizedKeysCommand`.** Держать миллионы ключей в одном файле `authorized_keys` нереально (линейный поиск). Поэтому sshd настраивается на **динамический lookup**:

```
AuthorizedKeysCommand /opt/gitlab-shell/authorized-keys-check git %u %k
AuthorizedKeysCommandUser git
```

sshd на каждый вход зовёт скрипт, тот спрашивает у платформы по API «есть такой ключ? чей он?» и отдаёт строку authorized_keys на лету. Так делают и GitLab, и Gitea — это снимает проблему гигантского файла ключей.

### Различия платформ

**GitLab — gitlab-shell.** Go-демон, который:

1. Принимает SSH-команду и идентичность ключа.
1. Спрашивает GitLab internal API (`/api/v4/internal/allowed`): можно ли этому пользователю это действие с этим репозиторием.
1. При успехе **проксирует** диалог к Gitaly через RPC `SSHUploadPack`/`SSHReceivePack`/`SSHUploadArchive`.

То есть gitlab-shell сам git не запускает — он гейтвей к Gitaly.

**Gitea — встроенный SSH-сервер.** Два режима:

- внешний `sshd` ОС + `AuthorizedKeysCommand`, указывающий на `gitea serv`;
- **встроенный SSH-сервер** на Go (библиотека `gliderlabs/ssh` поверх `golang.org/x/crypto/ssh`) — Gitea сама слушает порт, аутентифицирует ключ по своей БД и исполняет git-команду. Удобно в контейнерах: не нужен системный sshd.

**Bitbucket Server.** Встроенный SSH через **Apache MINA SSHD** (Java) — слушает отдельный порт, проверяет права через permission service, маршрутизирует к хранилищу (Mesh).

**GitHub.** Аналогично варианту A в гигантском масштабе: единый вход, идентичность по ключу, маршрутизация к Spokes-нодам, где лежат реплики репозитория.

**Certificate-based SSH.** Вместо регистрации каждого публичного ключа — доверие к CA, подписывающему пользовательские сертификаты (с TTL и principal). Поддерживается частью платформ (в т.ч. GitHub Enterprise/организациями) и сильно упрощает ротацию и offboarding: отозвал CA-доверие или дал короткий TTL — и не надо чистить миллионы ключей.

### Производительность: почему SSH быстрее HTTPS

- Бинарный канал без HTTP-заголовков и chunked-обёрток.
- Соединение устанавливается один раз на всю операцию (нет череды независимых POST как в stateless HTTP).
- Меньше «болтовни» round-trip на больших передачах.

Расплата — управление ключами и то, что SSH сложнее кэшировать/проксировать на CDN, чем HTTP.

### Граничные случаи

- **Скан публичного ключа на каждый вход** через `AuthorizedKeysCommand` нагружает API — отсюда кэширование результатов lookup.
- **PTY/agent-forwarding** должны быть отключены (`no-pty`, `no-agent-forwarding`), иначе вектор для злоупотреблений.
- **Огромный `authorized_keys`** (если кто-то не перешёл на command-lookup) → деградация времени логина.
- **Хостовые ключи при кластере**: ноды за одним адресом должны предъявлять согласованные host keys, иначе у клиентов сыплются предупреждения.

### Trade-off

- **command= в файле** против **динамического lookup**: простота против масштабируемости.
- **Встроенный SSH-сервер** (Gitea/Bitbucket) против **системного sshd**: меньше зависимостей и проще контейнеризация против зрелости и фич настоящего OpenSSH.
- **Сертификаты** против **ключей**: лёгкая ротация против необходимости поднять и охранять CA.

### Код

- `git/connect.c` — клиент: запуск ssh и формирование команды
- `git/Documentation/git-upload-pack.txt`, `git-receive-pack.txt`
- gitlab-shell: `cmd/`, `internal/command/` — диспетчер SSH-команд и проксирование в Gitaly
- Gitea: `cmd/serv.go` (внешний режим), `modules/ssh/ssh.go` (встроенный сервер)
- OpenSSH `sshd_config(5)`: `ForceCommand`, `AuthorizedKeysCommand`

## 2.3 HTTP Smart Protocol

Транспорт, который проходит через корпоративные прокси и фаерволы, поддаётся кэшированию и не требует управления SSH-ключами. По умолчанию для большинства публичных clone.

### Протокол — endpoints и заголовки

Две фазы = два запроса.

**Фаза 1 — info/refs (GET):**

```
GET /repo.git/info/refs?service=git-upload-pack
```

Ответ (smart):

```
Content-Type: application/x-git-upload-pack-advertisement

001e# service=git-upload-pack\n
0000
<затем ref advertisement в pkt-line, как в 2.1>
```

Первая pkt-line `# service=...` + flush — маркер, по которому клиент понимает, что сервер «умный» (smart), а не отдаёт статические файлы (dumb).

**Фаза 2 — собственно операция (POST):**

```
POST /repo.git/git-upload-pack
Content-Type: application/x-git-upload-pack-request
```

В теле — `want`/`have`/`done` (clone/fetch). Для push симметрично: `service=git-receive-pack`, POST на `/git-receive-pack`, `Content-Type: application/x-git-receive-pack-request`.

**Chunked transfer encoding.** Пак отдаётся потоком (`Transfer-Encoding: chunked`) — сервер начинает слать, не зная заранее итоговый размер. Это и позволяет «лить» большой clone.

**Dumb HTTP** (исторический, по сути мёртв): сервер просто раздаёт файлы из `.git` как статику, клиент сам ходит по объектам. Не требует серверной логики, но дико неэффективен (нет negotiation, нет паков на лету). Не использовать.

### Архитектура и различия платформ

**Базовый механизм — `git-http-backend`.** Это CGI из поставки Git, реализующий smart-HTTP. Минимальная связка:

```
nginx/Apache → git-http-backend (CGI) → upload-pack/receive-pack
```

**GitLab — Workhorse.** Тонкий Go-прокси **перед** Rails. Git-HTTP-запрос приходит в Workhorse, тот аутентифицирует через Rails (быстрый internal-вызов), но **тяжёлый стриминг пака** делает сам, минуя Ruby (Rails не должен качать гигабайты через себя), и проксирует к Gitaly. Workhorse появился именно чтобы снять с Rails большие загрузки/выгрузки.

**Gitea.** HTTP-роуты git обрабатывает встроенный сервер: аутентификация (Basic/token) в middleware, затем вызов git-процесса для negotiation/пака. Никакого внешнего CGI не требуется.

**Bitbucket.** HTTP-доступ к git идёт через приложение Atlassian, маршрутизирующее к хранилищу (Mesh); аутентификация — через свой механизм (App Passwords/токены).

**GitHub.** Огромный масштаб smart-HTTP с маршрутизацией к Spokes, кэшированием advertisement и bitmaps под капотом.

### Аутентификация по HTTP

- **Basic Auth**: логин + пароль/токен в заголовке. Пароль почти везде = PAT (см. 4.2), не «настоящий» пароль.
- **Token в URL** (`https://x-access-token:<token>@host/repo.git`) — удобно для CI, но утекает в логи/`.git/config`; осторожно.
- **Credential helpers** (`git-credential-store`, `osxkeychain`, `manager`) кэшируют учётки на клиенте, чтобы не вводить токен каждый раз.

### HTTP/2 и git

Smart-HTTP создавался под HTTP/1.1. HTTP/2 даёт мультиплексирование соединений, но git-операция — это по сути один большой стрим, поэтому выигрыш скромный. Поддержка зависит от фронт-прокси (nginx/ALB); на практике решает не версия HTTP, а близость сервера и bitmaps.

### Граничные случаи

- **Прокси, режущие chunked/буферизующие тело**: ломают стриминг больших clone (классическая причина «clone зависает на 99%»).
- **Лимиты тела запроса** (`client_max_body_size` в nginx) режут большие push по HTTP.
- **Таймауты фронт-прокси** короче времени упаковки большого репо → обрыв.
- **Stateless-налог**: длинная negotiation = много POST round-trip (см. 2.1), v2 это смягчает.
- **Token в `.git/config`** после clone-с-токеном-в-URL — частая утечка секретов.

### Trade-off

- **HTTP** против **SSH**: проходимость через прокси и лёгкость кэширования/балансировки против чуть большей латентности и сложной потоковой передачи.
- **Workhorse-подход** (вынести стриминг из апп-сервера) против простоты «один процесс делает всё»: производительность против числа компонентов.
- **Token-in-URL** против **credential helper**: удобство для скриптов против риска утечки.

### Код

- `git/http-backend.c` — эталонный smart-HTTP сервер (CGI)
- `git/remote-curl.c`, `git/http.c` — клиентская сторона HTTP-транспорта
- `git/Documentation/technical/http-protocol.txt` — первоисточник
- GitLab Workhorse: `gitlab.com/gitlab-org/gitlab/workhorse` (раньше отдельный репозиторий)
- Gitea: `routers/web/githttp.go` / `services/` — HTTP git endpoint

# Часть III. Хранилище и масштабирование

Здесь живёт суть нашей задачи: как превратить «git-репозитории на диске» в отказоустойчивый масштабируемый сервис. Четыре разных ответа индустрии на один вопрос.

## 3.1 Gitaly

Gitaly — это gRPC-сервис доступа к git-репозиториям, написанный на Go. Его придумал GitLab, чтобы убрать прямой доступ Rails к файловой системе и вынести git-операции на отдельный слой, который можно масштабировать, кэшировать и реплицировать.

### Архитектура — набор gRPC-сервисов

Репозитории на диске остаются обычными git-репозиториями. Gitaly даёт над ними API, разбитый по доменам (имена — ориентир, *сверяйте с актуальным `.proto`*):

```
RepositoryService   создание/удаление/проверка/архив/housekeeping репозитория
CommitService       чтение коммитов, log, история файла
BlobService         содержимое объектов
RefService          ветки, теги, перечисление refs
DiffService         диффы между коммитами
SmartHTTPService    upload-pack/receive-pack для HTTP-транспорта
SSHService          upload-pack/receive-pack/upload-archive для SSH
OperationService    мутации через серверный git (создать ветку, merge, cherry-pick, теги)
ObjectPoolService   пулы объектов для дедупликации форков
RemoteService       операции с удалёнными репозиториями
```

Принципиально: **тяжёлые мутации делаются через `OperationService`**, а не через «склей объект руками». Сервер сам запускает git с правильными хуками, проверками и транзакциями. Чтения (`Commit`/`Blob`/`Ref`/`Diff`) — отдельно и легче.

### Как Gitaly выполняет git-операции

Исторически было два пути: Go-нативная логика и `gitaly-ruby` (отдельный процесс для операций, изначально переиспользовавших Ruby-код GitLab). **gitaly-ruby со временем выпилен** — операции переведены на Go + прямой вызов git-бинарника (subprocess) или библиотечную работу с объектами. Для нас это значит: интегрируясь с Gitaly, мы получаем чистый Go-сервис, который под капотом во многом всё равно зовёт `git`.

### Аутентификация

Простой и эффективный механизм: **общий HMAC-токен** в gRPC-метаданных каждого запроса:

```
authorization: Bearer <gitaly_token>
```

Токен (`gitaly_token`) общий для клиента и Gitaly; запросы без валидного токена отбиваются. В проде поверх этого — **mTLS** между клиентом и Gitaly (взаимная проверка сертификатов), чтобы канал был и аутентифицирован, и зашифрован. Для нашей интеграции Gitea↔Gitaly это две настройки: токен в метаданных + (желательно) mTLS.

### Транзакции и хуки

**Reference transactions.** Когда обновляется ref, Gitaly запускает `reference-transaction` хук, который участвует в голосовании (через Praefect, см. 3.2). Это механизм, обеспечивающий, что все реплики договорились о новом значении ref до его фиксации.

**Хуки уровня git.** Gitaly устанавливает свои `pre-receive`, `update`, `post-receive`, `reference-transaction`. В мире GitLab эти хуки **звонят обратно** в GitLab API (через gitlab-shell), чтобы выполнить access checks (можно ли пушить в защищённую ветку) и сгенерировать события. **Это критическая точка нашей интеграции**: pre-receive должен звонить не в GitLab, а в Gitea — то есть наш слой обязан реализовать тот же callback-контракт (см. 4.1 и 8.1).

### Streaming и sidechannel

Большие ответы (вывод `git log`, содержимое больших блобов, пак при clone) **чанкуются** через gRPC streaming, а не возвращаются одним сообщением. Для отдачи **пака** есть **sidechannel**: тело пака идёт мимо обычного gRPC-сообщения по отдельному потоку, чтобы не платить накладные расходы протобуфа и копирований на гигабайтах. Если у нас clone больших репозиториев медленный — смотреть надо сюда.

### Прочее

- **pack-objects cache** — кэш результатов `upload-pack` (см. 7.1): несколько одинаковых clone подряд не пересобирают пак.
- **Feature flags** — Gitaly включает новое поведение через флаги. В мире GitLab ими управляет GitLab. В нашей интеграции нужно понимать, какие флаги влияют на используемые RPC, и фиксировать их состояние осознанно.
- **Housekeeping** — упаковка, оптимизация refs, построение commit-graph/bitmaps инициируются через RPC, а не cron’ом на хосте.

### Граничные случаи

- **pre-receive без callback** = защищённые ветки и проверки прав не работают. Самая частая ловушка кастомной интеграции.
- **Несовпадение версий git** на хосте Gitaly и ожиданий Gitaly: Gitaly чувствителен к версии git (использует свежие фичи).
- **gRPC message size limits**: забыл стриминг — упёрся в лимит размера сообщения на большом ответе.
- **Feature flag дрейф**: поведение RPC «вдруг поменялось» после обновления Gitaly из-за флага по умолчанию.

### Trade-off

- **gRPC-слой** против **прямого exec** (как в ванильной Gitea): управляемость, кэш, репликация против дополнительного сетевого хопа и сложности.
- **Серверные операции** (`OperationService`) против ручной сборки объектов: безопасность и единые проверки против гибкости.
- **Sidechannel** против чистого gRPC: производительность против усложнения транспорта.

### Код

- Gitaly: `proto/*.proto` — контракт всех сервисов (первое, что читать)
- `internal/gitaly/service/` — реализации сервисов
- `internal/git/` — обёртки над git
- `internal/gitaly/hook/` — хуки и callback-контракт
- Документация: `doc/` в репозитории Gitaly (design docs по транзакциям и sidechannel)

## 3.2 Praefect

Один Gitaly = одна точка отказа. Praefect — это прокси/маршрутизатор/менеджер репликации, который ставится **перед** несколькими Gitaly-нодами и делает из них отказоустойчивый кластер.

### Архитектура

```
Клиент (Rails / Gitea)
      │ gRPC
   Praefect ──────── PostgreSQL (метаданные: где какой репозиторий, какие реплики свежие)
   ╱   │   ╲
Gitaly Gitaly Gitaly   (физические ноды; каждая хранит реальные репозитории)
```

- **Virtual storage**: клиент видит одно логическое хранилище; Praefect отображает его на несколько физических Gitaly.
- **PostgreSQL** хранит метаданные: какие ноды содержат данный репозиторий, какая нода **primary** для него, и **generation numbers** (счётчики версий, по которым видно, кто отстал).

### Маршрутизация и primary

- Репозитории распределяются по нодам; у каждого репозитория есть **primary** (per-repository primary — primary назначается на уровне репозитория, а не всего кластера, чтобы нагрузка размазывалась).
- **Запись** идёт на primary и реплицируется на secondaries.
- **Чтение** Praefect раздаёт между **актуальными** репликами (read distribution) — отставшие в раздаче чтений не участвуют, чтобы не отдать устаревшие данные.

### Репликация: два механизма

**1. Reference transactions (сильная консистентность записи).** При мутации, меняющей refs, primary и secondaries **голосуют**: каждый считает хеш предполагаемого результата, Praefect собирает голоса. Изменение фиксируется только при кворуме согласных. Так предотвращается расхождение веток между репликами в момент записи.

**2. Replication jobs (асинхронное досыпание).** Не всё реплицируется синхронно. Praefect ставит в очередь задания репликации, чтобы догнать отставшие ноды (например, после того как нода была недоступна). Generation numbers показывают, кому что досылать.

> Нюанс: «strong consistency» в Praefect относится к согласованию ref-обновлений через голосование. Полная побайтовая идентичность всех реплик в любой момент не гарантируется — асинхронная репликация догоняет хвосты. Это нормальный инженерный компромисс, но его надо понимать при проектировании бэкапов и failover.

### Reconciliation и failover

- **Reconciler** периодически сверяет ноды и ставит задания репликации, чтобы устранить расхождения.
- **Failover**: если primary недоступен, Praefect может назначить новый primary из актуальных реплик. Бывает автоматический и ручной режим (зависит от конфигурации и версии).
- **Network partition**: при расколе сети между нодами кворум может не собраться — запись будет отвергаться, что предпочтительнее, чем разъехавшиеся реплики (выбор в пользу консистентности).

### Различия с другими подходами

- Это **GitLab-специфичный** стек. Praefect рассчитан на работу в связке с Gitaly и GitLab. Подружить его с Gitea напрямую — нетривиально (Praefect говорит на gRPC Gitaly, но ожидает определённого окружения и хуков).
- Аналог у Atlassian — **Mesh** (3.3), у GitHub — **Spokes/DGit** (3.4). Решают ту же задачу (HA + распределение), но архитектурно по-разному.

### Граничные случаи

- **Split-brain не допускается ценой доступности**: при partition запись встаёт.
- **Replication lag**: secondary отстал → его временно исключают из раздачи чтений (иначе клиент увидит старое состояние).
- **Рост таблиц Praefect в PostgreSQL** (очередь репликации) при затяжных проблемах нод.
- **Миграция репозитория между нодами** требует аккуратной координации, чтобы не потерять «свежесть» (generation).

### Trade-off

- **Сильная консистентность ref** (голосование) против латентности записи (ждём кворум).
- **Доступность против консистентности** при partition: Praefect выбирает консистентность.
- **Сложность кластера** (Praefect + PostgreSQL + N×Gitaly) против HA.

### Код

- Praefect: `internal/praefect/` в репозитории Gitaly
- `internal/praefect/datastore/` — метаданные, generation numbers, очередь репликации
- reference-transaction логика в `internal/gitaly/`
- Документация: design docs по Praefect и транзакциям в `doc/`

## 3.3 Bitbucket Mesh

Ответ Atlassian на ту же задачу. Раньше Bitbucket Data Center хранил все репозитории на общем **NFS** — это была единая точка контеншена и масштабируемости. Mesh — распределённое реплицируемое хранилище git, убирающее NFS.

> Внутренние детали Mesh документированы скромнее, чем Gitaly. Ниже — общая архитектурная картина; конкретику сверяйте с актуальной документацией Atlassian.

### Архитектура

- **Mesh-ноды** — отдельные сервисы хранения git, между которыми распределяются репозитории.
- Каждый репозиторий реплицируется на **несколько нод** (по умолчанию обычно три), запись подтверждается по **кворуму** — то есть достаточно, чтобы большинство реплик приняли запись.
- Приложение Bitbucket общается с Mesh-нодами по gRPC и маршрутизирует операции к нодам, держащим нужный репозиторий.

### Конкурентная запись и консистентность

- Кворумная запись означает: push считается успешным, когда большинство реплик зафиксировали изменение. Это даёт устойчивость к потере одной ноды без потери данных.
- Конкурентные push в один репозиторий сериализуются на уровне ноды-владельца/координатора, чтобы refs не разъехались.

### Mesh vs Gitaly+Praefect

|Аспект             |Gitaly + Praefect                             |Bitbucket Mesh                            |
|-------------------|----------------------------------------------|------------------------------------------|
|Координатор        |Отдельный Praefect + PostgreSQL для метаданных|Логика распределения внутри приложения/нод|
|Консистентность ref|Голосование reference-transactions            |Кворумная запись по репликам              |
|Метаданные кластера|Внешний PostgreSQL                            |Внутреннее состояние                      |
|Экосистема         |GitLab                                        |Bitbucket DC                              |

Обе системы решают одну проблему — HA и устранение единой ФС — но Praefect выносит координацию в отдельный сервис с внешней БД, а Mesh встраивает её в свой стек.

### Geo / распределение

Bitbucket Data Center использует Mesh как фундамент для распределения нагрузки и устойчивости в рамках кластера; вопросы гео-репликации решаются на уровне DC-архитектуры Atlassian (mirror-фермы для удалённых площадок и т. п.).

### Граничные случаи и trade-off

- **Миграция с NFS на Mesh** — отдельный проект (Atlassian описывает процедуру поэтапного переноса); нельзя «щёлкнуть переключателем».
- **Потеря кворума** при недоступности нескольких реплик блокирует запись (как и у Praefect — консистентность важнее доступности).
- Trade-off тот же класс: кворум даёт устойчивость ценой латентности и необходимости держать ≥3 реплик.

### Что изучать

- Документация Atlassian по Bitbucket Mesh (архитектура, требования к нодам, миграция)
- Сравнительные материалы Atlassian Engineering про отказ от NFS

## 3.4 GitHub: Spokes и DGit

Как GitHub хранит сотни миллионов репозиториев с высокой доступностью. Публично известная архитектура — **Spokes** (раньше называлась **DGit**, Distributed Git).

> По GitHub нет открытых исходников хранилища; ниже — то, что описано в инженерном блоге GitHub и публичных докладах.

### Идея — репликация на уровне git

- Каждый репозиторий хранится в **трёх репликах** (three-way), разнесённых по разным **failure domains** (разные стойки/зоны), чтобы потеря одной не приводила к потере данных и доступности.
- Репликация выполняется на **уровне git-операций**, а не блочного устройства: запись применяется к репликам согласованно (трёхфазный подход), чтение можно брать с любой актуальной реплики.
- Маршрутизатор направляет операцию к нодам, держащим реплики конкретного репозитория.

Эволюция: GitHub начинал с пар «активный/пассивный» на уровне ФС (DRBD-подобные решения), затем перешёл к DGit/Spokes — распределению на уровне git, что дало и масштабирование, и устойчивость.

### Object pools для форков

Та же идея, что в 1.1, но в проде огромного масштаба: тысячи форков популярного репозитория делят **общий пул объектов** через `alternates`, храня только уникальные объекты. GitHub перепаковывает и обслуживает пул на уровне fork-сети, чтобы bitmaps (1.2) покрывали популярные коммиты родителя и clone форка был мгновенным.

### Геораспределение и обслуживание

- Реплики разносятся по зонам; запросы маршрутизируются с учётом близости и здоровья реплик.
- Планировщик обслуживания (maintenance) автоматически запускает упаковку/оптимизацию репозиториев по сигналам (число loose-объектов, давность GC).
- Монорепозитории-гиганты (например, крупные проекты Microsoft) — отдельный класс нагрузки; под них работают partial clone, sparse-checkout и серверные оптимизации.

### Чему учит GitHub-архитектура

- **Репликация на git-уровне** (а не блочном) даёт согласованность операций и возможность читать с любой свежей реплики — концептуально это то же, что делают Praefect и Mesh.
- **Object pools** — главный приём против взрыва хранилища на форках.
- **Bitmaps + commit-graph** — главный приём против медленного clone/log на масштабе.

### Что изучать

- GitHub Engineering Blog: статьи про DGit/Spokes, про object pools, про ускорение clone битмапами
- Доклады GitHub про обслуживание монорепозиториев

## 3.5 Gitea

Наша основа. Понимать её пределы важнее всего — именно из них растёт решение прикрутить Gitaly.

### Архитектура хранилища

- Репозитории лежат на диске под корнем (`ROOT`, исторически `data/gitea-repositories` или настраиваемый путь) как обычные bare git-репозитории.
- Gitea вызывает **бинарник `git` через `os/exec`**: формирует команду, окружение (`GIT_*` переменные), запускает, парсит stdout. Слой тонкий — это и сила (прозрачность), и предел (вся нагрузка на локальную ФС и процессы).
- **LFS** хранится отдельно: локально, либо в объектном хранилище (S3/MinIO/Azure Blob) — это первый шаг выноса «тяжёлого» со стандартного диска.

### Хуки и проверки

Gitea ставит свои git-хуки, которые **звонят обратно в Gitea** (через приватный API / внутренний механизм, передавая контекст через `GITEA_*` окружение), чтобы проверить protected branches, права и сгенерировать события (webhooks, обновление UI). Это прямой аналог callback-контракта Gitaly↔GitLab — и именно его придётся воспроизвести при интеграции с Gitaly (8.1).

### Webhooks и очереди

Gitea доставляет webhooks через **внутреннюю очередь** с воркерами-горутинами. Очередь может быть персистентной (level-queue на диске) или на Redis — это даёт устойчивость доставки и переживание рестартов (см. 6.1).

### Clustering — где болит

- Нативно Gitea рассчитана на **одну ноду** с локальным диском. Горизонтальное масштабирование «из коробки» ограничено.
- Чтобы поднять несколько инстансов Gitea, нужна **общая ФС** под репозитории (NFS/CephFS) + общий PostgreSQL + общий Redis/очередь. Общая ФС возвращает ровно те проблемы контеншена, блокировок и консистентности, ради которых GitLab сделал Gitaly, а Atlassian — Mesh.
- **Forgejo** (форк Gitea) развивает часть этих направлений независимо — полезно поглядывать на их решения по масштабированию и федерации.

### Где Gitea начинает болеть

- **10K+ репозиториев на общей ФС**: latency git-операций растёт, housekeeping конкурирует за IO.
- **Пики параллельных clone**: каждый — отдельный git-процесс; CPU/IO упираются, нет sidechannel/кэша пака как в Gitaly.
- **Большие репозитории**: отсутствие битмап-оптимизаций уровня GitHub делает clone дороже.
- **HA**: одна нода с локальным диском — единая точка отказа; переход на общую ФС добавляет свой класс проблем.

Отсюда наш путь: **оставить Gitea как продукт/UI/мета-слой (PR, issues, права, webhooks), а хранение и git-операции отдать управляемому слою Gitaly** — получить кэш, sidechannel и (через Praefect или иную схему) репликацию, не теряя удобный интерфейс Gitea.

### Trade-off Gitea как платформы

- **Простота и лёгкость** (один бинарь, прозрачный exec git) против **пределов масштабирования** (одна нода / общая ФС).
- **Прямой контроль над git** против **отсутствия встроенного слоя HA/репликации**.

### Код

- Gitea: `modules/git/` — обёртки над `os/exec git`
- `services/repository/` — операции над репозиториями
- `modules/hook/` и хуковые обработчики — callback-контракт проверок
- `modules/queue/` — очереди (webhooks, задачи)
- `modules/storage/` — абстракция хранилища для LFS/attachments (локально/S3)

# Часть IV. Авторизация и контроль доступа

Git сам по себе не знает про «права». Вся авторизация — надстройка платформы, и почти вся она в итоге сводится к одной точке: момент обновления ref в `receive-pack`.

## 4.1 Pre-receive хуки и branch protection

### Протокол — как git даёт точку контроля

При push сервер запускает `git-receive-pack`, который вызывает хуки в строгом порядке:

1. **pre-receive** — один раз на весь push. На stdin получает по строке на каждый обновляемый ref:

```
<old-sha> <new-sha> <refname>
<old-sha> <new-sha> <refname>
```

Ненулевой exit code хука **отклоняет весь push целиком** (все refs). Это главный рубеж: здесь платформа решает «пускать или нет».

1. **update** — по разу на **каждый** ref, аргументы `<refname> <old-sha> <new-sha>`. Ненулевой код отклоняет **только этот** ref, остальные проходят.
1. **post-receive** — после успешной записи. Здесь генерируются события: webhooks, обновление UI, запуск CI, отправка уведомлений.

### Как из этого собирается branch protection

Все правила защиты — это логика внутри pre-receive/update:

- **Force push protection.** Force push = новый коммит **не является потомком** старого (не fast-forward). Проверка:

```bash
# если old достижим из new — это fast-forward (можно); иначе force push
git merge-base --is-ancestor <old-sha> <new-sha>
# exit 0 = ff, exit 1 = форс (отклонить, если ветка защищена)
```

- **Delete protection.** Удаление ветки = `new-sha` равен нулю (`0000000000...`). Отлавливается сравнением.
- **Create detection.** Создание = `old-sha` нулевой.
- **Required signed commits.** Хук проверяет наличие и валидность подписи (`git verify-commit` / разбор `gpgsig`) у новых коммитов.
- **CODEOWNERS / required reviews.** Сам push этого не знает — это enforcement на уровне merge (PR), но связан с теми же ветками. Файл CODEOWNERS парсится (glob-паттерн → владельцы), и платформа требует ревью от владельцев затронутых путей перед merge.
- **Merge queue** (GitHub/GitLab): чтобы избежать «зелёный по отдельности, красный вместе», изменения выстраиваются в очередь, проверяются в порядке применения к свежей базе и мёржатся последовательно.

### Различия платформ

- **GitHub.** Branch protection enforced в pre-receive на стороне файловых нод (Spokes), плюс проверки на уровне приложения для PR-правил.
- **GitLab.** Хуки Gitaly **звонят в GitLab internal API** (`/api/v4/internal/allowed`) через gitlab-shell/workhorse: можно ли этому пользователю это обновление ref. Решение принимает GitLab, Gitaly его применяет.
- **Bitbucket.** Branch permissions проверяются permission-сервисом при обновлении ref (поверх project- и repo-level прав).
- **Gitea.** Внутренний хук проверяет protected branch rules (кто может пушить, требуется ли PR, нужны ли проверки) через приватный API Gitea.

**Для нашей интеграции это центральный пункт.** Gitaly-хук по умолчанию ориентирован на callback в GitLab. Чтобы protected branches работали в связке Gitea+Gitaly, наш слой обязан реализовать эквивалент `/internal/allowed`: Gitaly-хук → наш endpoint → проверка прав в Gitea → разрешить/запретить. Если этого нет — push проходит без проверок прав (см. 8.1).

### Граничные случаи

- **Атомарность push**: pre-receive отклоняет всё или ничего; update — по ref. Путаница между ними ведёт к «половина refs прошла».
- **Гонка (TOCTOU)**: между проверкой и записью ветку успел подвинуть другой push. Спасает атомарный compare-and-swap значения ref (old должен совпасть в момент записи) — серверная fast-forward проверка.
- **Большой push** (тысячи refs за раз): pre-receive получает огромный stdin; наивный построчный вызов API на каждый ref = шторм запросов.
- **Хук-таймаут**: если callback в платформу висит, push висит. Нужны таймауты и быстрый путь отказа.

### Trade-off

- **pre-receive (всё-или-ничего)** против **update (по ref)**: атомарность против гранулярности.
- **Callback в приложение** (актуальные права, но сетевой хоп и связность) против **локальной логики в хуке** (быстро, но дублирует модель прав).

### Код

- `git/builtin/receive-pack.c` — порядок и вызов хуков
- `git/Documentation/githooks.txt` — контракт хуков (первоисточник)
- Gitaly: `internal/gitaly/hook/` — реализация хуков и callback
- gitlab-shell: обработчик `/internal/allowed`
- Gitea: `services/pull/`, `modules/hook/`, обработчики protected branch

## 4.2 Аутентификация и токены

Кто ты — отдельный вопрос от что тебе можно. Здесь — кто.

### Способы и их хранение

- **SSH-ключи.** На сервере хранится **публичный** ключ; вход — по доказательству владения приватным. Lookup по fingerprint (см. 2.2). Приватный ключ сервер не видит никогда.
- **Personal Access Token (PAT).** Хранится **в виде хеша** (платформа не должна уметь восстановить токен). У токена есть scope (что разрешено), срок жизни, иногда привязка к ресурсам. По HTTP передаётся как пароль в Basic Auth.
  - GitHub использует префиксы формата токена (`ghp_` для personal, `gho_` OAuth, `ghu_` user-to-server, `ghs_` server/installation) с base62-телом и **встроенной контрольной суммой** — это позволяет автоматическим сканерам секретов мгновенно распознавать утёкшие токены в публичном коде.
- **OAuth 2.0.** Для интеграций от имени пользователя:
  - **web flow** — редирект на платформу, согласие, обратный редирект с кодом, обмен кода на токен;
  - **device flow** — для устройств без браузера: показываем код, пользователь подтверждает на другом устройстве.
- **GitHub Apps vs OAuth Apps.** Важное различие:
  - **OAuth App** действует от имени пользователя, делит его rate limit.
  - **GitHub App** — самостоятельная сущность. Аутентифицируется так: приложение подписывает **JWT** своим приватным ключом → получает **installation access token** (короткий TTL, около часа) на конкретную установку. Права **гранулярные** (fine-grained permissions), rate limit выше и считается на установку. Это рекомендуемый способ для интеграций/ботов.
- **Deploy keys.** SSH-ключ, привязанный к **одному репозиторию** (read-only или read-write). Удобно для CI/деплоя без выдачи доступа ко всему аккаунту.
- **Bitbucket App Passwords** — аналог PAT со scope; **GitLab** — PAT, project/group access tokens, deploy tokens.
- **Gitea application tokens** — токены доступа со scope, хранятся хешированными; реализация в коде Gitea.

### Токен на клиенте

- **Credential helpers** (`store`, `cache`, `osxkeychain`, `manager`) кэшируют учётки, чтобы не вводить токен на каждую операцию.
- В CI токен часто кладут в переменную окружения или временный credential helper; **в URL — нежелательно** (утечка в логи и `.git/config`).

### Ротация без даунтайма

- Короткоживущие installation tokens (GitHub Apps) ротируются автоматически — обновляемся за TTL.
- Для долгоживущих PAT — практика перекрытия: создать новый, раскатить, потом отозвать старый.
- Сертификатный SSH (2.2) делает ротацию почти бесплатной: TTL + доверие к CA вместо ручной замены ключей.

### Граничные случаи

- **Утёкший токен в публичном репо**: отсюда контрольные суммы в формате токена и secret scanning.
- **Истёкший installation token посреди длинной операции** CI: нужно уметь перевыпустить и продолжить.
- **Скан публичного ключа на каждый SSH-вход** (2.2) нагружает API → кэш.
- **Слишком широкий scope** токена (классический «всё подряд») — нарушение наименьших привилегий.

### Trade-off

- **PAT** (просто) против **GitHub App** (сложнее настроить, но гранулярно, выше лимиты, безопаснее для ботов).
- **Долгоживущие токены** (удобно) против **короткоживущих** (безопаснее, но нужна авто-ротация).
- **Ключи** против **сертификатов**: zero-setup против лёгкого offboarding ценой поднятия CA.

### Код

- Gitea: `models/auth/`, `services/auth/`, обработчики токенов и OAuth
- GitHub Apps: документация по JWT → installation token (схема, не исходники)
- `git/credential.c`, `git/Documentation/gitcredentials.txt` — клиентские credential helpers

## 4.3 Изоляция репозиториев и multi-tenancy

Один сервис обслуживает много пользователей/организаций. Нельзя, чтобы один арендатор дотянулся до данных другого.

### Угрозы и защита

- **Path traversal через `.git`.** Имена путей и ссылок в репозитории нельзя слепо использовать в ФС-операциях. Git и платформы валидируют пути, запрещают выход за пределы каталога репозитория.
- **Symlink-атаки.** Записанный в дерево симлинк (mode `120000`), указывающий наружу, при наивной выкладке мог бы дать доступ к чужим файлам. Защита — не разыменовывать такие пути на сервере, контролировать checkout.
- **Repository discovery.** Git умеет «искать .git вверх по дереву». На сервере это опасно (можно случайно попасть в чужой/родительский репозиторий). Контролируется переменными:

```
GIT_CEILING_DIRECTORIES   куда не подниматься при поиске .git
GIT_DIR / GIT_WORK_TREE   явное указание границ
safe.directory            защита от выполнения git в чужом по владельцу каталоге (CVE-2022-24765)
```

`safe.directory` появился после реальной уязвимости: git, запущенный в каталоге, принадлежащем другому пользователю, мог исполнить чужой конфиг/хук.

- **Namespace isolation.** Один Gitaly обслуживает тысячи организаций; изоляция — на уровне путей хранилищ и неймспейсов refs внутри репозиториев. Gitea на одной ноде так же изолирует репозитории разных владельцев структурой каталогов.
- **Visibility (public/private/internal).** Где enforced: на уровне приложения **до** того, как пустить к git-операции. Видимость форка не наследуется автоматически от родителя при смене — это отдельная логика (приватный родитель ≠ автоматически приватные форки, и наоборот — частый источник путаницы и инцидентов).
- **LFS permissions.** LFS-объекты авторизуются **отдельно** от git-доступа (свой токен/проверка), потому что лежат в другом хранилище (S3/CDN). Забытая проверка здесь = приватные большие файлы доступны по прямой ссылке.
- **Submodule security.** Содержимое `.gitmodules` (URL сабмодулей) валидируется: вредоносный URL/протокол в сабмодуле — известный вектор (например, попытки заставить клиента ходить по `file://` или выполнять команды через ext-протоколы).

### Различия платформ

- **Bitbucket** добавляет **project-level** права поверх repo-level: разрешения проекта каскадируются на его репозитории.
- **GitHub/GitLab** — организации/группы с вложенными правами, командами, ролями.
- **Gitea** — организации, команды, права на уровне репозитория и организации.

### Граничные случаи

- **Смена visibility родителя форка**: что происходит с форками — требует явной политики.
- **Fork приватного репо**: данные не должны «протекать» в публичную fork-сеть/общий object pool.
- **LFS по прямой ссылке** в обход проверки прав.
- **Сабмодуль на «опасный» протокол**.

### Trade-off

- **Жёсткая изоляция** (отдельные каталоги/неймспейсы, много проверок) против **дедупликации** (общие object pools требуют аккуратности с приватностью).
- **Каскадные права** (удобно админить) против **точечных** (гранулярно, но дороже в управлении).

### Код

- `git/setup.c` — repository discovery, `safe.directory`, ceiling dirs
- `git/Documentation/git-config.txt` — `safe.directory`, `protocol.*.allow`
- Gitea: `models/perm/`, `services/repository/`, проверки visibility и прав
- LFS: `services/lfs/` в Gitea — отдельная авторизация LFS

# Часть V. Pull Requests и Code Review

PR — не часть git. Это концепция платформы, построенная поверх двух git-примитивов: refs и diff. Понимание этого снимает большинство «магии».

## 5.1 Pull Request как концепция поверх Git

### Протокол — PR хранится в служебных refs

Когда создаётся PR, платформа сохраняет указатели в специальных ref-неймспейсах:

- **GitHub:**
  
  ```
  refs/pull/<N>/head    ← вершина ветки-источника PR
  refs/pull/<N>/merge   ← результат пробного merge с базой (если мёржится без конфликта)
  ```
- **GitLab:**
  
  ```
  refs/merge-requests/<N>/head
  refs/merge-requests/<N>/merge
  ```
- **Bitbucket:**
  
  ```
  refs/pull-requests/<N>/from
  refs/pull-requests/<N>/merge
  ```

`head`-ref позволяет вытащить PR даже без доступа к форку-источнику (`git fetch origin pull/N/head`). `merge`-ref — это **предвычисленный пробный merge**: платформа периодически пытается слить источник с базой и сохраняет результат, чтобы показать «мёржится / конфликт» без работы в реальной ветке.

**Что происходит при force push в ветку PR:** `head`-ref переезжает на новый коммит, `merge`-ref пересчитывается, diff и обсуждение перепривязываются (см. ниже про якоря комментариев). Старые коммиты остаются достижимы через reflog/историю PR.

### Merge-стратегии и как они реализованы

- **Merge commit** (`--no-ff`): создаётся коммит с двумя родителями (база и источник). История нелинейна, но полностью сохранена.
- **Squash**: все коммиты PR схлопываются в один новый коммит поверх базы. История источника теряется, база линейна.
- **Rebase**: коммиты источника переписываются поверх базы (новые SHA, committer меняется, author сохраняется), затем fast-forward. Линейно, но SHA меняются.

Как генерируется merge-коммит: платформа задаёт authorship (часто — автор PR или мёржащий), timestamp, сообщение (шаблон), и两 родителя. Всё это делается серверным git (в Gitaly — через `OperationService`).

### Conflict detection без рабочей копии

Платформе не нужен checkout, чтобы понять, мёржится ли PR. Используется **серверный merge без рабочего дерева**:

```bash
git merge-tree <base> <source>     # вычисляет результат merge как дерево, сообщает конфликты
```

`merge-tree` (в современном режиме) отдаёт дерево-результат и список конфликтов чисто на объектах, не трогая worktree. Это позволяет асинхронно пересчитывать `merge`-ref и показывать статус мёржабельности.

### Прочие механики

- **Draft PR**: флаг состояния, блокирующий merge; технически тот же набор refs.
- **Auto-merge**: PR помечается «слить, когда выполнятся условия» (зелёный CI, нужные ревью); платформа ждёт событий и мёржит сама.
- **Merge queue** (5.1 ⇄ 4.1): сериализация мёржей через проверку на свежей базе.

### Различия платформ

- Имена неймспейсов (выше) — главное техническое различие.
- GitHub `merge`-ref и Checks API тесно связаны; GitLab разделяет pipeline’ы на detached и merged-result (см. 6.2); Bitbucket привязывает build status к PR.
- В нашей связке Gitea держит PR-метаданные (issues/PR — единая модель в Gitea), а git-операции мёржа уходят в Gitaly (`OperationService`).

### Граничные случаи

- **Force push в источник во время ревью**: перепривязка комментариев, пересчёт merge-ref, риск «протухших» апрувов (часть платформ сбрасывает апрув при новом push).
- **База уехала вперёд**: merge-ref устарел, нужно пересчитать; «мёржабельность» — величина во времени.
- **Конфликт только в merge, не виден в diff против базы**: почему нужен именно merge-tree, а не простой diff.
- **PR из форка** приватные данные/права: доступ к `head`-ref vs доступ к самому форку.

### Trade-off

- **Merge commit** (полная история) против **squash/rebase** (чистая линейная история ценой потери гранулярности/SHA).
- **Предвычисленный merge-ref** (быстрый UI) против стоимости его постоянного пересчёта.
- **Auto-merge/merge queue** (меньше ручной возни, нет «зелёный по отдельности — красный вместе») против сложности и задержек.

### Код

- `git/builtin/merge-tree.c` — серверный merge без worktree (conflict detection)
- `git/merge-ort.c` — современный движок merge (ort), считающий результат на объектах
- Gitaly: `proto/operations.proto` — `UserMergeBranch`, `UserRebase`, и т. п.
- Gitea: `services/pull/` — модель PR, мёрж-стратегии, пересчёт мёржабельности

## 5.2 Diff-алгоритмы и code review

Diff — второй примитив, на котором стоит ревью. Качество diff напрямую влияет на читаемость review.

### Протокол — алгоритмы сравнения

Git использует библиотеку **xdiff** (вариант libxdiff). Доступные алгоритмы (`git diff --diff-algorithm=`):

- **Myers** (по умолчанию) — алгоритм поиска кратчайшего редакционного скрипта, сложность примерно O(N·D), где D — размер различий. Минимизирует число изменённых строк. Иногда даёт «нечитаемые» диффы (съезжающие границы блоков).
- **Histogram** — улучшение patience; быстр и обычно даёт более «человеческие» диффы на коде (лучше выравнивает блоки). Часто рекомендуется как дефолт для исходников.
- **Patience** — сначала сопоставляет **редкие уникальные строки** как якоря, потом рекурсивно дополняет. Хорош, когда Myers «съезжает» (например, при перестановке функций) — даёт более интуитивный результат ценой скорости.
- **Minimal** — выжимает абсолютный минимум diff (дороже).

### Поверх diff: что делают платформы

- **Rename detection** (`-M`): git не хранит «переименование» — он видит «удалён файл A, добавлен файл B». Переименование **выводится** по similarity index (процент совпадения содержимого). Поэтому переименование с правками может показаться «удалил+добавил», если similarity ниже порога.
- **Unified vs split view**: платформа рендерит один и тот же diff либо построчно-объединённо, либо двумя колонками. Это представление, не разные диффы.
- **Word-diff / intra-line**: подсветка изменений внутри строки (`git diff --word-diff`) — отдельный проход по словам/символам внутри изменённых строк.
- **Suggest changes** (GitHub): ревьюер предлагает правку прямо в комментарии; платформа превращает её в патч и применяет к ветке коммитом. Под капотом — генерация и применение diff-патча.
- **Inline-комментарии и якоря.** Комментарий привязан к (файл, строка, сторона diff, коммит/diff-hunk). При новом push платформа пытается **перенести** якорь на соответствующую строку в обновлённом diff; если строка изменилась/исчезла — комментарий становится «outdated». Алгоритм переноса — сопоставление позиций между старым и новым diff.
- **Semantic diff**: попытки сравнивать на уровне AST (понимать, что «функция переехала», а не «удалена и добавлена»). Сложно в общем случае, языкозависимо; в мейнстриме чаще остаётся текстовый diff с хорошим алгоритмом.

### Различия платформ

- Дефолтный алгоритм и эвристики rename-detection настраиваются по-разному; восприятие «качества diff» во многом определяется именно выбором histogram vs Myers.
- Перенос комментариев при force push — место, где платформы заметно различаются по UX (часть переносит агрессивнее, часть сразу помечает outdated).

### Граничные случаи

- **Большие файлы** (100K+ строк): diff дорог; платформы сворачивают/обрезают/отказываются рендерить.
- **Бинарные файлы**: текстового diff нет; показывается «binary changed», хранятся обе версии (через объекты), сравнение — по хешу/размеру.
- **Переименование+правки** ниже порога similarity → «удалил+добавил» вместо «переименовал».
- **Сгенерированный код / vendored**: платформы помечают такие пути (`.gitattributes` `linguist-generated`), чтобы не зашумлять diff.
- **Перенос комментария** на «уехавшую» строку: риск показать комментарий не там.

### Trade-off

- **Myers** (минимум изменений, быстрый дефолт) против **histogram/patience** (читабельнее на коде, иногда медленнее).
- **Агрессивный rename detection** (удобно) против стоимости вычисления similarity на больших диффах.
- **Перенос комментариев** (сохранить контекст) против риска неверной привязки.

### Код

- `git/xdiff/` — реализация diff-алгоритмов (Myers, patience, histogram)
- `git/diffcore-rename.c` — rename detection по similarity
- `git/diff.c` — генерация unified diff, word-diff
- `git/Documentation/git-diff.txt`, `gitattributes.txt` (`linguist-*`, diff-атрибуты)
- Gitea: `services/gitdiff/` — парсинг diff для рендера и привязки комментариев

# Часть VI. CI/CD интеграция

Как git-события превращаются в запуски пайплайнов и как результаты сборок возвращаются обратно в PR и branch protection.

## 6.1 Webhooks — архитектура доставки

### Протокол — что и как отправляется

Webhook — это HTTP POST на заданный URL при событии (push, PR opened, comment, и т. д.). Тело — JSON с описанием события.

**Подпись.** Чтобы получатель убедился, что запрос от платформы, тело подписывается HMAC общим секретом:

```
X-Hub-Signature-256: sha256=<hmac_sha256(secret, body)>
```

Получатель пересчитывает HMAC по сырому телу и сравнивает. Важная деталь: сравнивать нужно по **сырым байтам** тела до парсинга (любая ре-сериализация JSON изменит подпись) и **constant-time** сравнением (защита от timing-атак).

**Заголовки события** (имена зависят от платформы): тип события, уникальный delivery id (для идемпотентности и реплея).

### Гарантии доставки

- **At-least-once.** Платформа гарантирует «доставим хотя бы раз», но **не «ровно один раз»**. Получатель обязан быть идемпотентным (использовать delivery id для дедупликации).
- **Retry с backoff.** При ошибке/таймауте получателя — повторы с нарастающей задержкой, до предела попыток.
- **Порядок не гарантирован.** События могут прийти не в том порядке, в котором произошли (параллельная доставка, ретраи). Нельзя полагаться на порядок — состояние надо восстанавливать из payload, а не из последовательности.
- **Fan-out.** Один push может породить много webhook-доставок (несколько подписчиков, несколько хуков).

### Различия платформ

- **GitHub.** Очередь доставки с ретраями, история доставок и **replay** (можно переотправить вручную), at-least-once.
- **GitLab.** Доставка через **Sidekiq** (Ruby background jobs) — очередь с воркерами и ретраями.
- **Gitea.** Внутренняя **очередь** (level-queue на диске или Redis) + воркеры-горутины; переживает рестарт, ретраит. Реализация в `modules/queue` + `services/webhook`.
- **Bitbucket.** Доставка через событийную платформу Atlassian.

### Observability и отладка

- **История доставок**: что отправили, какой код вернул получатель, тело запроса/ответа.
- **Replay**: переотправить событие при отладке.
- Логирование: код ответа, латентность, число ретраев.

### Граничные случаи

- **Получатель лежит**: ретраи копятся; нужен предел и dead-letter, иначе очередь распухает.
- **Медленный получатель**: держит воркеры, бьёт по пропускной способности доставки → таймауты.
- **Неидемпотентный обработчик** + at-least-once = двойная обработка (двойной деплой!). Решается dedup по delivery id.
- **Подпись по распарсенному JSON** вместо сырого тела = вечный mismatch.
- **Большой payload**: лимиты размера; иногда платформа шлёт «тонкое» событие, за деталями — отдельный API-запрос.

### Trade-off

- **At-least-once + ретраи** (надёжно) против необходимости идемпотентности на стороне получателя.
- **Персистентная очередь** (переживает рестарт) против накладных расходов на хранение.
- **Параллельная доставка** (пропускная способность) против гарантий порядка (их нет).

### Код

- Gitea: `services/webhook/`, `modules/queue/` — очередь и доставка
- GitLab: webhook + Sidekiq jobs
- Документация платформ: формат payload, заголовки, схема подписи

## 6.2 Statuses, Checks и CI-интеграция

Как результат сборки возвращается в git-платформу и блокирует/разрешает merge.

### Протокол — Commit Status API

Базовый, общий для всех механизм. CI постит статус на конкретный коммит:

```
POST /repos/<owner>/<repo>/statuses/<sha>
{
  "state": "pending" | "success" | "failure" | "error",
  "context": "ci/build",          // имя проверки (уникальный ключ)
  "target_url": "https://ci/.../123",  // ссылка на лог
  "description": "Build #123"
}
```

Платформа хранит последний статус на пару (sha, context). Branch protection требует, чтобы перечисленные **required status checks** были `success`, иначе merge заблокирован.

### Check Runs (GitHub) — богаче статусов

GitHub Apps умеют **Check Runs**: помимо состояния — **аннотации** (привязка замечаний к строкам файла прямо в diff), вывод, действия (кнопки «re-run», «fix»), детальные шаги. Это отдельный, более мощный слой, чем простой Status. Conclusion: success/failure/neutral/cancelled/timed_out/action_required.

### Pipeline-модели платформ

- **GitLab Pipelines.** `.gitlab-ci.yml` в репозитории описывает stages/jobs. Пайплайн привязан к коммиту и/или MR. Различают:
  - **detached MR pipeline** — гоняется на коммите ветки MR;
  - **merged result pipeline** — гоняется на **результате пробного merge** с целевой веткой (ловит конфликты интеграции заранее);
  - **merge train** — аналог merge queue: последовательная проверка на свежей базе.
- **Bitbucket Pipelines.** `bitbucket-pipelines.yml`; результаты сборки возвращаются в PR через Build Status.
- **Gitea Actions.** Совместимы с синтаксисом **GitHub Actions** workflow (`.gitea/workflows/*.yml` или `.github/workflows`), исполняются раннером **act_runner** (на базе `nektos/act`). Это даёт переиспользование экосистемы Actions при self-hosted Gitea.

### Self-hosted runners — протокол

- Раннер **регистрируется** в платформе (токен регистрации) и затем **поллит** задачи (long-poll/очередь): «есть работа для меня?».
- Получив job, исполняет в изолированном окружении (контейнер/VM), стримит логи, по завершении репортит статус/conclusion и заливает **артефакты**.
- Артефакты хранятся платформой с TTL и лимитом размера (связь с хранилищем — отдельная подсистема, часто объектное хранилище/S3).

### Гонки и тонкости

- **Pending-гонка**: между push и регистрацией пайплайна есть окно. Если branch protection требует проверку, нужно гарантировать, что статус `pending` появится до попытки merge (иначе merge может «проскочить» или зависнуть в ожидании несуществующей проверки).
- **Статус на «уехавший» SHA**: статус привязан к sha; после rebase/amend старые статусы не относятся к новому коммиту.

### Граничные случаи

- **Required check, который ни разу не пришёл**: PR навсегда заблокирован ожиданием — частая операционная боль.
- **Флапающие проверки** держат merge queue.
- **Артефакты переполняют хранилище**: нужен TTL и квоты.
- **Раннер-призрак**: зарегистрирован, но не поллит — задачи висят.

### Trade-off

- **Commit Status** (простой, универсальный) против **Check Runs** (богатый UX: аннотации/действия, но GitHub-специфичный и через Apps).
- **merged-result pipeline** (ловит конфликты интеграции) против лишних запусков и сложности.
- **Self-hosted runners** (контроль, доступ к внутренним сетям) против эксплуатации и безопасности раннеров.

### Код

- Gitea: `services/actions/`, интеграция act_runner; модель commit status в `models/`
- GitLab: `.gitlab-ci.yml` парсер, pipeline-модель
- Документация: Commit Status API, GitHub Checks API (схема), синтаксис workflow

# Часть VII. Производительность и observability

Как сделать так, чтобы на масштабе было быстро, и как увидеть, когда станет медленно.

## 7.1 Кэширование

### Что и где кэшируется

- **commit-graph** (1.2) — главный кэш для обхода истории: предвычисленные родители, даты и **generation numbers**. Ускоряет `git log`, `merge-base`, подсчёт ahead/behind на порядки, потому что не нужно парсить каждый commit-объект. Должен регулярно обновляться (после крупных push).
- **Reachability bitmaps** (1.2) — кэш достижимости для быстрого clone/fetch: «что нужно отправить» читается из битмапа, а не вычисляется обходом графа. Ключ к быстрому clone у GitHub.
- **pack-objects cache (Gitaly)** — кэширование результата `git pack-objects` при отдаче (upload-pack). Несколько одинаковых clone подряд (типично для CI, который клонирует один и тот же коммит на множестве раннеров) переиспользуют готовый пак вместо пересборки. Очень полезно под CI-нагрузкой.
- **HTTP-кэширование advertisement**: ответ `info/refs` для не изменившегося репозитория можно кэшировать (ETag/условные запросы) — экономит на массовых анонимных clone.
- **Кэш статистики репозитория**: число коммитов, контрибьюторов, размер — дорого считать каждый раз, кэшируется и инвалидируется по push.
- **Blame-кэш**: `git blame` дорог на больших/«горячих» файлах; платформы кэшируют результат.
- **Поисковый индекс кода**: отдельная подсистема (Elasticsearch; у GitHub — **Zoekt** для code search). Индексирует содержимое, обновляется по push. Это не git-кэш, а параллельный индекс.
- **Redis (GitLab)**: сессии, очередь (Sidekiq), кэш приложения. У Gitea Redis опционален (кэш/очередь/сессии).
- **CDN для LFS**: большие файлы раздаются через CDN, разгружая ядро платформы; авторизация — отдельная (4.3).

### Инвалидация при push

После записи нужно согласованно сбросить/обновить зависимые кэши: статистику репозитория, кэш веток/тегов, при необходимости перестроить commit-graph/bitmaps (обычно отложенно, в housekeeping), переиндексировать код. Порядок важен: UI не должен показывать новый коммит при устаревшей статистике слишком долго.

### Различия платформ

- **GitHub** — максимум на bitmaps/commit-graph и собственном планировщике обслуживания; Zoekt для поиска.
- **Gitaly** — pack-objects cache + housekeeping, строящий commit-graph/bitmaps.
- **Gitea** — стандартный git (commit-graph через housekeeping), опциональный Redis, LFS в объектном хранилище.

### Граничные случаи / trade-off

- **Устаревший commit-graph** после большого push → временно медленный log, пока не пересобран.
- **Холодный pack-objects cache** на первом clone (выгода — на повторных).
- **Инвалидация vs консистентность**: агрессивный сброс кэшей бьёт по производительности, ленивый — показывает устаревшее.
- **Поисковый индекс** отстаёт от push (eventual): код найдётся не мгновенно.

### Код

- `git/commit-graph.c`, `git/pack-bitmap.c`
- Gitaly: pack-objects cache в `internal/`
- Zoekt (github/zoekt) — code search индекс

## 7.2 Мониторинг и observability

Нельзя эксплуатировать то, что не видишь. Для git-сервиса ключевые сигналы — латентность git-операций, ошибки RPC и насыщение ресурсов.

### Ключевые метрики Gitaly

(имена — ориентир, *сверяйте с актуальным экспортером*)

- `grpc_server_handled_total{grpc_method,grpc_code}` — число обработанных RPC по методам и кодам → **error rate** по методам.
- `grpc_server_handling_seconds` (гистограмма) → **latency p50/p95/p99** по RPC.
- `gitaly_command_*` (cpu/real seconds, число запусков git-подпроцессов) → сколько и как долго мы реально гоняем git.
- Метрики pack-objects cache (hit/miss) → эффективность кэша clone.

**Что алертить:**

- рост p99 латентности ключевых RPC (upload-pack, операции мутаций);
- рост error rate (не-OK gRPC коды);
- насыщение: CPU, IO-wait, дескрипторы, длина очередей;
- глубина/возраст очереди задач.

### Praefect

- **replication lag** и глубина очереди репликации → насколько secondaries отстают.
- число **unhealthy nodes**, статусы primary/secondary.
- объём таблиц метаданных в PostgreSQL (распухание очереди при проблемах).

### Трассировка и логи

- **Связать HTTP/SSH-запрос с git-подпроцессом**: distributed tracing (OpenTelemetry/Jaeger) через всю цепочку (фронт → приложение → Gitaly → git). GitLab использует трассировку, чтобы видеть, где в цепочке теряется время.
- **Slow-log** для git-операций: что считать «медленным» (порог по латентности/CPU), логировать с контекстом (репозиторий, RPC, размер).
- **Структурные логи** с correlation id, чтобы склеивать события одного запроса.

### Методологии

- **RED** (Rate, Errors, Duration) — для RPC-сервиса вроде Gitaly: запросы/сек, доля ошибок, длительность.
- **USE** (Utilization, Saturation, Errors) — для ресурсов нод: загрузка CPU/диска, насыщение (очереди, IO-wait), ошибки железа.

### Capacity planning

- Тренды: рост числа репозиториев, объёма на диске, IOPS, латентности под нагрузкой.
- Предсказывать, когда кончится **место** (рост репозиториев + LFS) и **IOPS** (пики параллельных clone/CI).
- Бенчмарки clone больших репозиториев по регионам/нодам.

### Различия платформ

- **GitHub** измеряет clone latency по регионам, имеет глубокую внутреннюю телеметрию.
- **GitLab** — Prometheus-метрики Gitaly/Praefect + распределённая трассировка.
- **Gitea** — встроенный набор метрик скромнее; для прода добавляют экспортеры/обвязку. Bitbucket — JMX/Atlassian-аналитика.

### Граничные случаи / trade-off

- **«Зависший push»**: чаще всего — висящий callback хука (4.1), исчерпание дескрипторов, или GC под нагрузкой; видно только при сквозной трассировке.
- **Слишком подробные логи/трейсы** бьют по производительности и стоят денег на хранение → сэмплирование.
- **Алерты без SLO** → шум; нужны пороги, привязанные к целям (например, p99 upload-pack < X мс).

### Код / инструменты

- Gitaly: `/metrics` Prometheus endpoint; конфиг логирования/трейсинга
- Praefect: метрики репликации и здоровья нод
- OpenTelemetry/Jaeger, Prometheus + Grafana, экспортеры для Gitea

# Часть VIII. Наша архитектура: Gitea + Gitaly

Здесь сходится всё предыдущее. Мы взяли Gitea как продукт (UI, PR/issues, права, webhooks, организации/команды) и заменили её «голый exec git по ФС» на управляемый слой Gitaly. Эта часть — карта того, что нужно контролировать, и чек-листы для команды.

> Этот раздел описывает **контракты и зоны риска** интеграции, а не конкретную нашу реализацию. Сверяйте с нашим кодом и фиксируйте принятые решения рядом с каждым пунктом.

## 8.1 Кастомная интеграция Gitea → Gitaly

### Карта: какие RPC мы используем и под что

|Поведение Gitea              |RPC Gitaly (ориентир)                       |Заметка                        |
|-----------------------------|--------------------------------------------|-------------------------------|
|Создание/удаление репо       |`RepositoryService` (Create/Remove/Exists)  |проверять идемпотентность      |
|Просмотр кода/истории в UI   |`CommitService`, `BlobService`, `RefService`|стриминговые ответы            |
|Diff в PR                    |`DiffService`                               |связать с рендером 5.2         |
|clone/fetch/push по HTTP     |`SmartHTTPService`                          |sidechannel для пака (2.1, 3.1)|
|clone/fetch/push по SSH      |`SSHService`                                |upload/receive/upload-archive  |
|Мёрж PR, создание веток/тегов|`OperationService`                          |серверный git + хуки/транзакции|
|Дедупликация форков          |`ObjectPoolService`                         |если используем пулы           |

**Чек-лист для команды:** напротив каждой строки зафиксировать — используем ли, какую версию `.proto`, какие именно методы.

### Аутентификация канала

- **HMAC-токен** (`gitaly_token`) в gRPC-метаданных каждого запроса (3.1). Хранить как секрет, ротировать.
- **mTLS** между Gitea и Gitaly в проде: взаимные сертификаты, шифрование канала. Без него токен ходит по незащищённому каналу.
- Чек: где лежит токен, кто его ротирует, включён ли mTLS, как обновляются сертификаты.

### Контракт хуков — критический пункт

В мире GitLab хуки Gitaly **звонят в GitLab API** (`/api/v4/internal/allowed`) для access checks (4.1). У нас этого GitLab нет. Значит **наш слой обязан реализовать эквивалент**:

```
push → Gitaly receive-pack → pre-receive хук → НАШ endpoint (вместо /internal/allowed)
     → проверка прав/protected branches в Gitea → allow/deny
     → (при allow) запись refs → post-receive → события в Gitea (webhooks, UI, CI)
```

**Если этот контракт не реализован полностью:**

- protected branches не работают (любой может пушить в `main`);
- force-push/delete protection не срабатывает;
- события (webhooks, обновление UI, запуск Actions) не генерируются после push.

**Чек-лист:**

- [ ] pre-receive вызывает наш endpoint и умеет отклонять push
- [ ] реализованы force-push detection (`merge-base --is-ancestor`) и delete detection (нулевой new-sha)
- [ ] проверка прав пользователя на ветку идёт через модель прав Gitea
- [ ] post-receive генерирует webhooks и обновляет состояние в Gitea
- [ ] reference-transaction хук подключён (если используем Praefect)
- [ ] таймауты на callback (висящий хук = висящий push, 4.1)

### Streaming и производительность

- Все потенциально большие ответы (`log`, `blame`, blob, пак) — через **gRPC streaming**, иначе упрёмся в лимит размера сообщения (3.1).
- Отдача пака — через **sidechannel** (2.1), иначе clone больших репо платит налог протобуфа.
- Чек: проверить, что на больших репозиториях clone использует sidechannel; замерить p99 upload-pack.

### Feature flags

- Gitaly включает поведение через флаги; в нашей интеграции **нет GitLab, который ими управляет**. Нужно осознанно зафиксировать состояние флагов, влияющих на используемые RPC, и перепроверять при обновлении Gitaly (3.1).
- Чек: список значимых флагов и их зафиксированные значения; процедура ревью флагов при апгрейде.

### Репликация (если включаем Praefect)

- Praefect рассчитан на связку с Gitaly; поставить его перед нашими Gitaly-нодами реально, но нужно обеспечить контракт reference-transactions и хуков (3.2).
- Альтернативы Praefect для нашего случая: репликация на уровне ФС (CephFS) под Gitaly, либо своя схема. Каждая со своими гарантиями консистентности.
- Чек: какой режим консистентности нам нужен (строгая для refs?), сколько реплик, как failover.

### Известные зоны риска интеграции

- **Версия git** на хостах Gitaly должна соответствовать ожиданиям Gitaly (3.1).
- **Несоответствие версий** Gitea-клиента и Gitaly-сервера по `.proto` (ломается при апгрейде одной стороны).
- **Хук-контракт** — самая частая «тихая» поломка (push проходит, но без проверок/событий).
- **Изоляция и пути** (4.3): при своём слое легко случайно ослабить namespace-изоляцию.
- **LFS** (4.3): авторизуется отдельно; не забыть в нашей схеме.

## 8.2 Операционные аспекты кластера

### Rolling update без даунтайма

- Обновлять Gitaly-ноды по одной; при наличии Praefect — выводить ноду из ротации, обновлять, возвращать, ждать догона репликации.
- Следить за совместимостью `.proto` Gitea↔Gitaly на время смешанных версий.
- Чек: порядок обновления (Gitaly → Praefect → Gitea или иной), проверка здоровья после каждого шага.

### Добавление новой Gitaly-ноды

- Зарегистрировать в Praefect (метаданные в PostgreSQL), дождаться, пока reconciler/replication догонит на неё нужные репозитории (3.2).
- Чек: процедура ввода ноды, контроль generation numbers/lag до перевода чтений на неё.

### Rebalancing репозиториев

- Перенос репозитория между нодами требует аккуратности с «свежестью» (generation), чтобы не отдать устаревшую реплику (3.2).
- Чек: как мигрируем, как проверяем целостность после переноса (`git fsck`).

### Backup и консистентность

- Бэкап git-репозитория должен быть **консистентным**: либо через серверный механизм (Gitaly/GitLab backup создаёт бандлы), либо снапшот ФС в момент без активной записи. Простой `cp` «горячего» репо может поймать полузаписанный pack/ref.
- Бэкапить нужно: репозитории (Gitaly), метаданные Praefect (PostgreSQL), БД Gitea (PostgreSQL), конфиги/секреты, LFS-хранилище.
- Чек: что бэкапим, как часто, как проверяем восстановимость (тестовый restore!).

### Disaster recovery — RTO/RPO

- **RPO** (сколько данных допустимо потерять): зависит от режима репликации. Строгая консистентность refs (голосование) минимизирует потерю refs; асинхронный хвост репликации может стоить последних мутаций на упавшей ноде (3.2).
- **RTO** (за сколько поднимемся): зависит от наличия горячих реплик и автоматизации failover.
- Чек: целевые RTO/RPO записаны и проверены учениями.

### Тестирование failover

- Регулярно прогонять failover в **staging**: убить primary, убедиться, что новый назначается, чтения/записи продолжаются, данные согласованы.
- Чек: сценарии (потеря ноды, partition сети, отказ PostgreSQL Praefect), ожидаемое поведение, runbook.

### Disk и housekeeping

- Запускать упаковку/оптимизацию (commit-graph, bitmaps, упаковка refs) через RPC housekeeping Gitaly, **не** cron-git на хосте (3.1, 7.1) — иначе можно поймать гонку с RPC-слоем.
- Следить за cruft packs и grace-периодом (1.2): мусор может держать место.
- Чек: расписание housekeeping, мониторинг роста loose-объектов и места.

### Логи и retention

- Ротация и retention логов Gitaly/Praefect; сэмплирование трейсов (7.2), чтобы не утонуть в данных и стоимости.

### Дебаг «push завис»

Типовой алгоритм (по 4.1, 7.2):

1. Трейс запроса: где встал — на хуке (callback в Gitea), на git-подпроцессе, на репликации?
1. Проверить callback-endpoint: отвечает ли, не таймаутит ли.
1. Дескрипторы/ресурсы Gitaly-ноды (USE).
1. Не идёт ли тяжёлый housekeeping/GC параллельно.
1. Состояние Praefect: не ждём ли кворум при partition (3.2).

### Capacity planning

- Сколько репозиториев на ноду — определять замерами (latency под нагрузкой), а не «на глаз» (7.2).
- Планировать IOPS под пики параллельных clone (CI) и место под рост репо+LFS.
