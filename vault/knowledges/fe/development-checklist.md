# Чеклист разработки фронтенда

Источник: [VCS-10805](https://portal.works.prod.sbt/pages/viewpage.action?pageId=...) — Процесс разработки нового фронта

---

## 1. Обработка ошибок

- ✅ Не показывать технические детали (stack trace, raw error object)
- ✅ Использовать понятные пользователю сообщения об ошибках
- ✅ Предоставлять кнопку "Повторить" (retry) при уместности

---

## 2. Обработка состояния загрузки

- ✅ Показывать спиннер/заглушку при загрузке данных
- ✅ Не оставлять экран пустым во время загрузки

---

## 3. Пустое состояние (Empty State)

**Описание:** Если API вернул пустой результат (`[]`), показывать соответствующее UI-состояние: иконку, текст-подсказку, CTA-кнопку.

**Типы пустоты:**
- Нет данных вообще → "У вас пока нет задач"
- Нет совпадений по фильтру → "Ничего не найдено. Попробуйте изменить параметры"

**Рекомендации:**
- Не оставлять экран "пустым" — это вызывает неуверенность
- Учитывать разные типы пустоты (как выше)

---

## 4. Состояние с контентом

**Описание:** При наличии данных отрисовка должна быть стабильной, производительной и соответствовать дизайн-системе.

**Рекомендации:**
- Безопасно обрабатывать все поля (`undefined`, `null`)
- Поддерживать lazy-loading и виртуализацию для больших списков
- Для сущностей со статусами доступа (`private/internal/archive/source_hub`) использовать визуально различимые маркеры (badge/tokens), а не одинаковые теги

---

## 4.1 Соответствие макету (пиксельные ограничения)

**Описание:** Если в FE-design/макете зафиксированы конкретные размеры, это часть требований, а не рекомендация.

**Проверить перед PR:**
- ширина и позиционирование контейнера страницы;
- ключевые внутренние отступы;
- минимальная высота карточек/элементов списка;
- поведение на согласованных брейкпоинтах.

**Рекомендации:**
- фиксировать цифры в `fe/design.md`, а не только в обсуждении;
- не добавлять новые контролы без подтверждения в SA/FE-артефактах.

---

## 5. Права доступа (Access)

**Описание:** UI должен адаптироваться под уровень доступа текущего пользователя. Используются ручки `/access/repo` или `/access/project`.

**Рекомендации:**
- Не полагаться только на UI-ограничения — права всегда проверяются на бэкенде
- Отсутствие прав не должно вызывать ошибки рендера

---

## 6. Проверка отрисовки при разных правах доступа

**Проверить:**
- Нет "мерцающих" элементов (появление/исчезновение кнопок)
- Все состояния (пусто, ошибка, контент) корректны для каждой роли

**Рекомендации:**
- Автоматизировать через Storybook с разными mock-ролями
- Использовать feature flags или dev-tools для переключения ролей

---

## 7. data-testid

**Назначение:** Упрощают написание E2E- и компонентных тестов.

**Рекомендации:**
- Минимум: на ключевые интерактивные элементы (кнопки, формы, таблицы, модалки)
- Семантические имена: `data-testid="project-create-repo-button"`
- Уникальность внутри компонента
- Обязательно добавлять для элементов с условным рендером

---

## 8. WireMock

**Обязанности:**
- Отражать актуальную структуру и поведение бэкенд-API
- Покрывать все состояния: success, error, empty, loading

**Рекомендации:**
- После изменений API — обновлять моки
- Использовать моки для поиска и фильтрации списков
- Для фильтров проверять, что в запрос уходят только контрактные параметры; параметры без UI-контрола (если так задумано) проверять через URL-state сценарии

---

## 9. Ссылки между старым и новым фронтендом

**Правило:** Избегать `full-page reload` при переходе между частями нового фронта.

---

## 10. Standalone режим

**Описание:** Микрофронтенд должен корректно работать как в составе основного приложения (MFE), так и в изолированном режиме.

**Рекомендации:**
- Проверять работу в standalone-режиме на стенде
- Протестировать standalone-сборку перед мержем

---

## 11. Ссылки LegacyUI vs ReactApp

### LinkRouter
- **Для:** переходов внутри ReactApp
- **Результат:** бесшовный переход через React Router

### Link (из sds)
- **Для:** перехода на LegacyUI (GoTemplates)
- **Результат:** hard link с перезагрузкой страницы
- **Зачем:** Нужен для инициализации GiteaBridgeMFE
