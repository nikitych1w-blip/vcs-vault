# Как встроить коллекцию в процесс тестирования?

---
## Что такое коллекция?

Коллекция — это тестовая модель SourceControl, расположенная в `docs/test-model/`.
Она представляет собой иерархическое дерево заметок Obsidian, где каждый узел соответствует одной тестируемой области (странице, функции, объекту).

Точка входа: [[vault/test-model/INDEX]] (`docs/test-model/INDEX.md`)
Терминология: [[dictionary]] (`docs/test-model/FAQ/dictionary.md`)

---
## Структура коллекции

Иерархия строится по уровням вложенности, закреплённым тегами `#lv0`–`#lv8`:

| Уровень | Тег           | Пример                                        |
| ------- | ------------- | --------------------------------------------- |
| 0       | `#lv0`        | SourceControl (корень)                        |
| 1       | `#lv1`        | API, CLI, UI, Общие механизмы, Ролевая модель |
| 2       | `#lv2`        | Проект, Репозитории, Дашборд, Аутентификация  |
| 3+      | `#lv3`–`#lv8` | Вложенные экраны, функции, сценарии           |
Каждый узел дерева — это `.md`-файл с набором тегов. Теги определяют контекст узла (продукт, интерфейс, сущность).

---
## Шаг 1 — Открыть коллекцию в Obsidian

1. Запустить Obsidian.
2. Открыть vault: `docs/test-model`.
3. Открыть **Graph View** — узлы графа соответствуют файлам, рёбра — ссылкам между ними.
4. Точка входа — файл `INDEX.md` или `SourceControl.md`.
---
## Шаг 2 — Выбрать область для тестирования

Найти нужный раздел одним из способов:

**По индексу**
Открыть `INDEX.md` → перейти по ссылке до нужного уровня вложенности.

**По тегу**
В Obsidian: панель **Tags** → выбрать тег. Примеры:
- `#repo` — все узлы, связанные с репозиторием
- `#merge #request` — все сценарии запросов на слияние
- `#settings` — настройки (проекта или репозитория)
- `#lv3` — только третий уровень вложенности

**По поиску**
`Ctrl+Shift+F` → ввести термин (используй словарь `dictionary.md` для правильных формулировок).

---
## Шаг 3 — Интерпретировать узел как тест-кейс

Каждый файл-узел описывает один тестируемый объект. Теги файла задают контекст:

```
#lv5 #SC #UI #FE #project #repo #merge #request
```

Это означает: **UI → Проект → Репозиторий → Запросы на слияние** (уровень 5).

Из набора тегов формируется:
- **Путь** — навигационная цепочка от корня до узла
- **Область** — какой интерфейс тестируется (`#UI`, `#api`, `#cli`)
- **Сущность** — что тестируется (`#repo`, `#branch`, `#commit` и т. д.)
---
## Шаг 4 — Покрыть ветви нижнего уровня

Для каждого узла с дочерними файлами:

1. Открыть файл родительского узла.
2. Перейти в **Backlinks** или **Outgoing links** — увидеть все дочерние узлы.
3. Убедиться, что каждый дочерний узел (`#lv+1`) покрыт тест-кейсом.

Листовые узлы (нет дочерних) — это минимальная единица тестирования.

---
## Шаг 5 — Связать узлы с тест-кейсами в TMS

В каждый файл-узел можно добавить ссылку на тест-кейс в TMS:

```markdown
<!-- TMS: TC-1234 -->
```

Или использовать поле frontmatter:

```yaml
---
tags:
  - SC
  - lv5
  - repo
  - merge
tms: TC-1234
status: covered   # covered | partial | missing
---
```

Статусы покрытия:
- `covered` — тест-кейс написан и актуален
- `partial` — покрыт частично
- `missing` — тест-кейс отсутствует

---
## Шаг 6 — Контролировать покрытие

Использовать Obsidian **Dataview** (если установлен плагин) для выборки непокрытых узлов:

```dataview
TABLE tms, status
FROM "docs/test-model"
WHERE status = "missing" OR status = "partial"
SORT file.path ASC
```

Без Dataview — ручная проверка через поиск по тегу `#lv` + фильтр по `status: missing`.
