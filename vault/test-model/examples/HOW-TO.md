# Как добавлять узлы в тест-модель

Гайд отвечает на вопрос: **появилась новая функциональность — куда и как добавить её в систему координат**.

---

## Принципы системы

Структура директорий = иерархия узлов. Каждая папка — узел. Внутри папки — файл с тем же именем и тегами.

```
5. Настройки/
└── Настройки.md   ← содержит теги узла
```

Числовой префикс в имени папки = уровень узла (lv).

---

## Шаг 1 — Найди родителя

Открой карту сущностей в `skills/common/model.md` или INDEX.md и ответь на вопрос:
**внутри какого существующего узла появилась новая функциональность?**

Пример: на странице репозитория появилась страница управления деплой-ключами.

Страница репозитория = узел `UI / Проект / Репозитории / Репозиторий` (lv4).
Настройки репозитория = `5. Настройки` (lv5) — именно там живут подобные вещи.

Родитель: `1. UI / 2. Проект / 3. Репозитории / 4. Репозиторий / 5. Настройки`

---

## Шаг 2 — Определи уровень нового узла

Уровень нового узла = уровень родителя + 1.

```
Родитель — lv5 (Настройки)
Новый узел — lv6 (Деплой-ключи)
```

Если внутри нового узла есть отдельные операции (Добавить, Удалить, Просмотреть) — они lv7.
Если у операции есть атомарные сценарии (валидация формата, лимит количества) — они lv8.

---

## Шаг 3 — Создай директорию и файл

Соглашение по имени: `<lv>. <Название>/`

```
5. Настройки/
└── 6. Деплой-ключи/          ← новая папка
    ├── Деплой-ключи.md        ← файл с тегами
    ├── 7. Добавить/
    │   └── Добавить.md
    ├── 7. Удалить/
    │   └── Удалить.md
    └── 7. Просмотреть/
        └── Просмотреть.md
```

---

## Шаг 4 — Заполни файл узла

Минимальный frontmatter + теги:

```markdown
---
tags:
  - SC
  - UI
  - lv6
  - repo
  - settings
status: missing
---
```

**Обязательные теги:**
- `#SC` — всегда
- интерфейс: `#UI` / `#api` / `#CLI`
- уровень: `#lv<N>`
- сущность: `#repo`, `#branch`, `#commit` и т.д. — если применимо
- `#ПАО` — если функциональность критичная и требует приоритетного покрытия

**Статус покрытия** — всегда `missing` при создании.

---

## Шаг 5 — Обнови INDEX.md

Добавь ссылку на новый узел в нужное место индекса:

```markdown
## 1. UI
- [[UI]]
  - [[Проект]]
    - [[Репозитории]]
      - [[Репозиторий]]
        - [[Настройки]]
          - [[Деплой-ключи]]   ← добавить
```

---

## Шаг 6 — Обнови карту сущностей

В `skills/common/model.md` добавь узел в дерево:

```
│       └── Настройки              lv5
│           └── Деплой-ключи       lv6–lv7   ← добавить
```

---

## Полный пример: страница управления деплой-ключами

**Ситуация:** в репозитории появился раздел «Деплой-ключи» в настройках. Можно добавить SSH-ключ с правами на чтение или запись, просмотреть список, удалить.

### Путь в модели

```
UI / Проект / Репозитории / Репозиторий / Настройки / Деплой-ключи
```

### Создаваемые узлы

| Путь | lv | Теги |
|------|----|------|
| `5. Настройки/6. Деплой-ключи/` | lv6 | `#SC #UI #lv6 #repo #settings` |
| `6. Деплой-ключи/7. Добавить/` | lv7 | `#SC #UI #lv7 #repo` |
| `6. Деплой-ключи/7. Удалить/` | lv7 | `#SC #UI #lv7 #repo` |
| `6. Деплой-ключи/7. Список/` | lv7 | `#SC #UI #lv7 #repo` |
| `7. Добавить/8. Валидация ключа/` | lv8 | `#SC #UI #lv8 #repo` |
| `7. Добавить/8. Права доступа/` | lv8 | `#SC #UI #lv8 #repo` |

### Файл `Деплой-ключи.md`

```markdown
---
tags:
  - SC
  - UI
  - lv6
  - repo
  - settings
status: missing
---
```

---

## Когда нужны API-узлы

Если функциональность доступна через API — создай параллельный узел в `1. API/`:

```
1. API/
└── 2. api v3/
    └── v3.md  ← добавь эндпоинт в описание или создай отдельный узел
```

Если это отдельная группа эндпоинтов — заведи подпапку по аналогии с UI.

---

## Чек-лист при добавлении узла

- [ ] Родительский узел найден в `skills/common/model.md`
- [ ] Уровень (lv) = уровень родителя + 1
- [ ] Имя папки: `<lv>. <Название>/`
- [ ] Файл внутри: `<Название>.md` с тегами и `status: missing`
- [ ] Добавлена запись в `INDEX.md`
- [ ] Обновлена карта в `skills/common/model.md`
- [ ] Параллельный API-узел создан (если нужен)
- [ ] QA уведомлён о новом узле для планирования TC
